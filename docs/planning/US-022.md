## Analisi requisiti

La storia chiede due cose distinte che vanno tenute separate: un identificatore **interno** (per lo sviluppatore, derivato da commit hash/data, "chiaro a me ma non all'utente") e un numero **user-facing** semplice. Mappando gli AC: il numero deve essere deterministico per commit (stesso commit → stessa versione, sempre), distinguibile tra commit diversi (per confrontare prod vs test), generato automaticamente ad ogni build (zero step manuali), e visibile in due punti UI (login, dashboard) con lo stesso valore.

Edge case rilevanti: build eseguita da un working tree con modifiche non committate (git status dirty) — la versione deve comunque generarsi, basandosi sull'ultimo commit disponibile (non blocca la build); build eseguita senza repository git disponibile (es. ambiente CI che scarica solo i sorgenti senza `.git`) — va previsto un fallback placeholder per non rompere la build. Requisito implicito: la versione deve essere coerente tra login e dashboard nello stesso deploy (stesso valore, non calcolato due volte in due punti diversi con margine di disallineamento).

Ambiguità risolta con assunzione esplicita (nessuna domanda bloccante, sotto soglia delle 2 ambiguità critiche): la storia riguarda solo le pagine **frontend** (login, dashboard) — la versione del **backend** (.NET API) resta fuori scopo per questa storia, come già segnalato nelle Open questions del backlog. Se in futuro servirà anche su API, sarà una storia separata.

## Soluzione tecnica

Versione generata **a build-time** del frontend Angular, da uno script Node eseguito prima di `ng build`, che legge `git rev-list --count HEAD` (contatore monotono crescente = numero user-facing, es. `v247`) e lo short hash del commit (`git rev-parse --short HEAD`, es. `a1b2c3d`, usato come dato "interno" leggibile solo da chi guarda il `title` HTML/tooltip, non mostrato in chiaro nel testo principale). Lo script scrive un file TypeScript `src/app/version.ts` con due costanti, importato da un componente condiviso.

**Decisioni chiave:**
- **Contatore commit vs data/hash puro:** scelto il contatore (`git rev-list --count`) come numero principale perché è l'unico tra le opzioni che è insieme leggibile, monotono e univoco per commit — la data da sola non distingue due commit nello stesso giorno, l'hash puro non è "semplice da comprendere" per l'utente.
- **Componente condiviso vs duplicazione:** creato `AppVersionComponent` standalone riutilizzato in login e dashboard, invece di duplicare la logica due volte — garantisce stesso valore nei due punti per costruzione, ed è la singola unità testabile con uno unit test.
- **Generazione via script wrapper nei deploy script esistenti, non via npm `pre`-hook automatico:** lo script `scripts/generate-version.js` viene invocato esplicitamente da `scripts/deploy-frontend.ps1` (già usato per build+zip, vedi sessione corrente) prima di `npm run build` / `ng build --configuration testing`. Evita comportamento "magico" implicito di npm lifecycle hooks (`prebuild`) che non scatterebbe comunque su invocazioni dirette di `ng build`.
- **`version.ts` committato con placeholder, sovrascritto ad ogni build reale:** evita di rompere `ng serve` in dev (nessun generator richiesto per lo sviluppo locale) al costo di un file che può apparire "sporco" nel diff se qualcuno fa build locale e commit per errore — mitigato aggiungendolo a `.gitignore` dopo il commit iniziale del placeholder (pattern già usato in questo repo per `appsettings.*.json`).

**Componenti toccati:**

| Componente | Tipo modifica |
|---|---|

**Modifiche al modello dati:** nessuna (nessun DB coinvolto, è puro frontend build-time).

**Contratti API:** nessuno (fuori scopo backend, vedi Analisi requisiti).

**Rischi specifici:**
- Ambiente di build senza `.git` disponibile (raro in questo setup, dato che si builda sempre dalla working copy locale prima di zippare) → lo script deve fare fallback a placeholder `'dev'` invece di crashare la build.
- Se in futuro la build avviene su una macchina diversa dal repo locale (es. CI/CD vero), il contatore commit richiede history completa (`git rev-list`) — un checkout shallow (`--depth=1`) romperebbe il conteggio. Non è il caso oggi (build sempre da clone locale completo) ma va annotato.

## Task

**TASK-1** — Script generatore versione
- File: `frontend/golp-app/scripts/generate-version.js` (nuovo, Node.js, nessuna dipendenza esterna oltre `child_process`/`fs`)
- Logica: esegue `git rev-list --count HEAD` → `APP_VERSION = 'v' + count`; esegue `git rev-parse --short HEAD` → `APP_BUILD_HASH`; se uno dei due comandi fallisce (no `.git`), usa fallback `APP_VERSION = 'v0-dev'`, `APP_BUILD_HASH = 'unknown'`; scrive `src/app/version.ts` con `export const APP_VERSION = '...'; export const APP_BUILD_HASH = '...';`
- Acceptance: eseguito due volte sullo stesso commit produce lo stesso identico contenuto file; eseguito su commit diversi produce `APP_VERSION` diverso
- Verifica: `node scripts/generate-version.js && cat src/app/version.ts` (manuale)

**TASK-2** — Placeholder committato
- File: `frontend/golp-app/src/app/version.ts` (nuovo, valori placeholder `'v0-dev'` / `'unknown'`)
- Acceptance: `ng serve` funziona senza errori di import anche senza aver mai eseguito il generator
- Verifica: clone pulito, `npm start`, nessun errore in console

**TASK-3** — Componente condiviso `AppVersionComponent`
- File: `frontend/golp-app/src/app/shared/version/app-version.component.ts` (nuovo, standalone, importa le costanti da `version.ts`, template inline minimale: `<footer class="app-version" [title]="buildHash">{{ version }}</footer>`)
- Acceptance: componente esporta `version` e `buildHash` come campi pubblici leggibili dal template; nessuna logica oltre import diretto delle costanti
- Verifica: unit test in TASK-4

**TASK-4 (test)** — Unit test `AppVersionComponent`
- File: `frontend/golp-app/src/app/shared/version/app-version.component.spec.ts` (nuovo, Karma/Jasmine)
- Asserzioni: il componente renderizza il testo di `APP_VERSION` nel DOM; l'attributo `title` dell'elemento footer contiene `APP_BUILD_HASH`
- Verifica: `npm test` (watch singolo file o suite intera)

**TASK-5** — Integrazione in login
- File: `frontend/golp-app/src/app/auth/login/login.component.html` (aggiunge `<app-version>` dopo `auth-footer`), `login.component.ts` (aggiunge `AppVersionComponent` agli `imports`)
- Acceptance: AC1 (login mostra versione) soddisfatto visivamente
- Verifica: manuale, `ng serve` → pagina login

**TASK-6** — Integrazione in dashboard
- File: `frontend/golp-app/src/app/dashboard/dashboard.component.ts` (aggiunge `AppVersionComponent` agli `imports`, aggiunge `<app-version></app-version>` nel template inline dopo i link)
- Acceptance: AC2 (dashboard mostra versione, stesso valore del login) soddisfatto
- Verifica: manuale, confronto visivo valore identico a quello mostrato in login nello stesso build

**TASK-7** — Wiring nei deploy script esistenti
- File: `scripts/deploy-frontend.ps1` (aggiunge `node scripts/generate-version.js` dentro `Push-Location $appDir` prima delle chiamate `npm run build` / `ng build --configuration testing`)
- Acceptance: AC3 (generazione automatica, nessun intervento manuale ad ogni build) soddisfatto per il flusso di deploy reale già in uso
- Verifica: eseguire `./scripts/deploy-frontend.ps1 -Env Production`, ispezionare `version.ts` rigenerato nello zip prodotto

**TASK-8** — `.gitignore` per `version.ts` generato
- File: `.gitignore` (aggiunge riga `frontend/golp-app/src/app/version.ts` dopo il commit iniziale del placeholder in TASK-2)
- Acceptance: build locali successive non producono diff rumorosi su questo file
- Verifica: `git status` dopo una build locale non mostra `version.ts` come modificato

**TASK-9 (test manuale, end-to-end)** — Verifica determinismo cross-commit
- Acceptance AC4/AC5: stesso commit → stessa versione in due build successive; commit diversi → versioni diverse
- Procedura: build su commit A, annotare versione mostrata; build su commit B (dopo un commit qualsiasi), annotare versione; confermare che A≠B e che rifare il build su A ripristina lo stesso valore di prima
- Verifica: manuale, confronto valori a schermo tra login (ambiente prod) e login (ambiente test) dopo deploy da commit diversi — utile anche per lo use case originale ("verificare se due ambienti sono sincronizzati")

## Test strategy

| AC | Test |
|---|---|
| Login mostra numero di versione | TASK-5 (manuale) + TASK-4 (unit, componente condiviso renderizza il valore) |
| Dashboard mostra stesso numero di versione | TASK-6 (manuale) + TASK-4 (stesso componente, stessa garanzia per costruzione) |
| Generazione automatica ad ogni build, no intervento manuale | TASK-7 (manuale, verifica negli script di deploy esistenti) |
| Stesso commit → stessa versione (deterministico) | TASK-1 (manuale, doppia esecuzione) + TASK-9 (e2e manuale cross-build) |
| Commit diversi → versioni diverse (verifica sync ambienti) | TASK-9 (manuale, cross-commit) |
| Formato semplice, senza esporre hash/data direttamente | TASK-3/TASK-4 (unit — verifica che il testo visibile sia `APP_VERSION`, non `APP_BUILD_HASH`; l'hash resta solo nel `title`) |

**Strategia e2e:** non serve Playwright dedicato — è una stringa statica nel DOM, già coperta da unit test sul componente condiviso più verifica manuale visiva nei due punti. Aggiungere un e2e qui sarebbe overkill per il valore della storia (LOW priority, 3 punti).

**Demo video:** non necessaria. La storia produce una piccola label di testo, non un flusso interattivo — uno screenshot delle due pagine (login + dashboard) con la versione visibile è sufficiente come evidenza, se richiesto in fase di review.

## Rischi aperti
- Backend non espone una propria versione in questa storia — se in futuro serve confrontare anche la versione API deployata (non solo frontend), andrà aperta una storia dedicata (possibile follow-up naturale, non in scope qui).
- Il meccanismo dipende da `git` disponibile e history completa nella macchina che esegue la build — vale per il flusso di deploy locale attuale (vedi sessione di deploy in corso), da rivalutare se in futuro si introduce un CI/CD con checkout shallow.
