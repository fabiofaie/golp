## Analisi requisiti

L'endpoint cross-circolo è il pezzo nuovo: query su `Matches` con join a `Circle` (per `Name`, `Sport`) e `MatchSets` (per score), filtrata sui 4 slot player dell'utente corrente. Il campo `myTeam` è derivato (1 se userId è in Team1, 2 altrimenti). Il campo `myDelta` usa la stessa logica già in `GetMatchesAsync`. La paginazione è server-side: `?page=1&pageSize=20` + `totalCount` in risposta. Il filtro `status` è opzionale: assente = Tutte, `pending` o `disputed` per i toggle specifici. `hasCurrentUserConfirmed` richiede join con `MatchConfirmations`.

La dashboard è attualmente un componente inline senza servizi propri — va estratto un `MyMatchHistoryComponent` standalone per non gonfiare il componente radice. Il proxy `/match` già esiste in `proxy.conf.json`, quindi `GET /match/mine` è raggiungibile senza modifiche al proxy.

Edge case: utente senza partite → risposta `{ totalCount: 0, page: 1, items: [] }` (mai 204, mai array nudo — serve struttura paginata per il FE). Partite Quick Match incluse automaticamente perché la query filtra per userId sui 4 slot senza distinzione di tipo-circolo.

## Soluzione tecnica

Nuovo file `MyMatchEndpoints.cs` con group `/match/mine` (distinto da `/match/quick`), registrato in `Program.cs` dopo `MapQuickMatchEndpoints`. La query EF usa `.Include(m => m.Sets).Include(m => m.Circle)` filtrato per userId nei 4 slot, ordinato `CreatedAt desc`, paginato con `.Skip().Take()` + `CountAsync` separato. `MatchConfirmations` caricato in un dizionario per `hasCurrentUserConfirmed`.

Lato FE: `MyMatchesPageComponent` è una pagina routed standalone (`/my-matches`), non embedded nella dashboard. `DashboardComponent` riceve solo un link/pulsante "Le mie partite" → `/my-matches`. La pagina gestisce stato locale: `items[]`, `filter`, `page`, `totalCount`, `loading`. Al cambio filtro: reset `page=1` e `items=[]`. Load-more: appende senza reset.
