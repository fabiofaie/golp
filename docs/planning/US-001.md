## Analisi requisiti

**Flussi principali:** tre flussi distinti — registrazione, login, recupero password. La registrazione crea l'account e autentica subito (nessun passaggio email di verifica, MVP). Il login non rivela quale campo è sbagliato (errore generico) per evitare user enumeration. Il recupero password usa un link via email con token a scadenza 1 ora; il vecchio token di sessione viene invalidato lato client al completamento.

**Edge case critici:** (1) email enumeration nel recovery — il backend risponde identicamente se l'email esiste o meno; (2) token reset multipli — solo l'ultimo è valido, i precedenti vengono invalidati all'emissione; (3) token brute force — token generato con 32 byte crittograficamente casuali, salvato come SHA-256 hash in DB. Open question "verifica email alla registrazione" risolta come NO per MVP: l'account è attivo subito.

**Requisiti impliciti:** HTTPS obbligatorio (JWT in transito), password hashing con BCrypt, servizio email astraibile (stub in development, implementazione configurabile in production), CORS configurato per dominio Angular. JWT claims: `sub` (userId), `email`, `exp` (1h). Nessun claim circolo — viene aggiunto nelle storie successive.

## Soluzione tecnica

**Approccio:** endpoints auth raggruppati sotto `/auth/*` nel Minimal API. Angular `AuthModule` autonomo con guard e interceptor. Password reset basato su token hash: plain token nell'email, `SHA-256(token)` in DB — confronto hash-to-hash. JWT in `localStorage` con scadenza breve (1h): scelta pragmatica per PWA su dominio separato (CORS + HttpOnly cookie complesso); rischio XSS mitigato da sanitization Angular e token expiry.

**Componenti toccati:**

| Componente | Tipo | Descrizione |
|---|---|---|
| `Users` | DB table | id, name, email (unique), password_hash, created_at |
| `PasswordResetTokens` | DB table | id, user_id, token_hash, expires_at, used_at |
| `AppDbContext` | EF Core | DbSet<User>, DbSet<PasswordResetToken> |
| `JwtService` | Backend service | Genera + valida JWT HS256, configurabile via appsettings |
| `PasswordHasher` | Backend util | BCrypt wrap |
| `PasswordResetService` | Backend service | Genera token 32-byte, salva SHA-256 hash, invia email |
| `IEmailService` | Backend interface | Astrazione SMTP/SendGrid; stub in development |
| `AuthEndpoints` | Minimal API | POST /auth/register, /auth/login, /auth/password-reset/request, /auth/password-reset/confirm |
| `AuthModule` | Angular | Register, Login, ForgotPassword, ResetPassword |
| `AuthService` | Angular service | HTTP calls + gestione JWT in localStorage |
| `AuthGuard` | Angular | Protegge route private, redirect a /login |
| `AuthInterceptor` | Angular | Aggiunge `Authorization: Bearer <token>` a ogni request |

**Modello dati:**

```sql
CREATE TABLE Users (
  Id          UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
  Name        NVARCHAR(100) NOT NULL,
  Email       NVARCHAR(254) NOT NULL UNIQUE,
  PasswordHash NVARCHAR(100) NOT NULL,
  CreatedAt   DATETIME2 NOT NULL DEFAULT GETUTCDATE()
);

CREATE TABLE PasswordResetTokens (
  Id        UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
  UserId    UNIQUEIDENTIFIER NOT NULL REFERENCES Users(Id),
  TokenHash NVARCHAR(64) NOT NULL,
  ExpiresAt DATETIME2 NOT NULL,
  UsedAt    DATETIME2 NULL
);
```

**Contratti API:**

```
POST /auth/register
  Body: { name, email, password }
  200: { token: "jwt..." }
  400: { error: "..." }   // password troppo corta, campi mancanti
  409: { error: "..." }   // email già registrata

POST /auth/login
  Body: { email, password }
  200: { token: "jwt..." }
  401: { error: "Credenziali non valide" }   // messaggio identico per email o pw sbagliata

POST /auth/password-reset/request
  Body: { email }
  200: { }   // sempre 200, non rivela se email esiste

POST /auth/password-reset/confirm
  Body: { token, newPassword }
  200: { }
  400: { error: "..." }   // token scaduto, già usato, password troppo corta
```

**Rischi:**
- Email delivery in production: Azure VM potrebbe avere porte SMTP bloccate. Valutare SendGrid free tier (100 email/giorno) come implementazione concreta dell'`IEmailService`.
- Invalidazione JWT dopo reset: MVP usa token breve (1h) + logout client-side. Nessuna blacklist JWT in DB — accettabile per MVP.
