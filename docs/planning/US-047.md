## Analisi requisiti

La feature abilita la modalità singolo (1v1) per gli sport che la supportano, controllata da un flag `AllowsSingles` per sport e da un flag `IsSingles` per partita. La scelta di ELO unificato (singolo + doppio nella stessa classifica) è deliberata: non va aggiunto un secondo ranking o filtro per formato.

Il cambio più invasivo è `Team1Player2Id`/`Team2Player2Id` che diventano `Guid?` nell'entity `Match` — tutti i punti che costruiscono l'array `playerIds` da questi 4 campi devono filtrare i null prima di usarli (lookup utenti, check partecipante, token pubblici, ELO). Le partite esistenti (tutte 2v2) sono backward-compatible: `IsSingles=false`, player2Id non-null.

La soglia di conferme è l'unico "magic number" presente nel codice: `totalCount == 4` in `PrepareConfirmAsync` diventa `match.IsSingles ? 2 : 4`. Stessa logica si applica al check di partecipazione (`playerIds.Contains(userId)`) che ora deve operare su un array filtrato dai null.

Edge case rilevanti: guest player in singolo funziona senza modifiche (il meccanismo `ResolveOrCreateGuestAsync` è slot-based); K-factor count query controlla tutti e 4 i campi player, i null in singles restituiscono false sul filtro LINQ → nessun falso positivo.

---

## Soluzione tecnica

**Approccio:** modifiche chirurgiche a 5 file di produzione + 1 migrazione + test. Nessun nuovo servizio. `ComputeDeltas` (statico, usato dai test esistenti) resta immutato: il branching singolo/doppio vive interamente in `CalculateAndApplyAsync`. La biforcazione è concentrata in due zone: validazione in `CreateMatchAsync` e calcolo ELO in `RatingService`.

**Componenti toccati:**

| File | Cambiamento |
|---|---|
| `Data/Entities/Sport.cs` | + `AllowsSingles bool` |
| `Data/Entities/Match.cs` | `Team1Player2Id`/`Team2Player2Id` → `Guid?`; + `IsSingles bool` (default false) |
| `Data/AppDbContext.cs` | seed: `AllowsSingles=true` per padel e beachtennis |
| `Services/ISportsService.cs` | `SportDto` record + `AllowsSingles` |
| `Services/SportsService.cs` | `.Select(…)` include `AllowsSingles` |
| `Endpoints/MatchEndpoints.cs` | CreateMatch: lookup sport, branch validation, IsSingles sul match; playerIds: null-safe in tutti gli endpoint; PrepareConfirmAsync: threshold dinamico |
| `Services/RatingService.cs` | `CalculateAndApplyAsync`: branch `IsSingles` (2 player ID, no media, 2 delta) |
| `Migrations/` | `AddSinglesSupport` |
| `Golp.Tests/Integration/` | nuovi test case (vedi Test strategy) |

**Contratto API (delta):**
- `GET /sports` → `SportDto` aggiunge campo `allowsSingles: bool`
- `POST /circles/{id}/matches` → body aggiunge `isSingles?: bool` (default false)
- `GET /circles/{id}/matches` + `GET …/{matchId}` → `team1`/`team2` ora hanno 1 o 2 elementi in base a `isSingles`

**Decisione chiave — ComputeDeltas invariato:**  
`ComputeDeltas` è statico e puro, testato direttamente. Per singolo, `CalculateAndApplyAsync` chiama la versione a 2 elementi: `ComputeDeltas(r1, r2, [k1, k2], team1Won, scoreRatio)` e usa solo `deltas[0]`/`deltas[1]`. Alternativa (overload o refactoring) è più pulita a lungo termine ma eccede lo scope.

## Rischi aperti

- `ComputeDeltas` accetta `int[] kValues` di dimensione variabile ma attualmente restituisce sempre 4 elementi (hard-coded nel return). Per singolo si chiama con 2 elementi e si usano solo `deltas[0]`/`deltas[1]` — funziona, ma il ritorno ha ancora 4 slot (gli altri due corrispondono a kValues fittizi se non passati). **Fix:** passare esattamente 2 kValues e usare i primi 2 del return. Controllare che `ComputeDeltas` non esegua accesso a indice 3 quando `kValues.Length == 2` → attualmente sì (riga `DeltaFor(kValues[3], …)`). Quindi serve un overload o refactoring minore per evitare `IndexOutOfRangeException`.  
  **Soluzione nel TASK-8:** non chiamare `ComputeDeltas` direttamente per singolo — calcolare i 2 delta inline con la stessa formula `DeltaFor`, oppure fare overload separato.
