## Analisi requisiti

**Mappa AC → comportamento concreto:**

| AC | Comportamento | Constraint |
|----|---------------|-----------|
| AC1 | `GET /circles/{cId}/matches/pending` → lista partite pending dove user è uno dei 4 giocatori | Scoped per circolo, solo partite `pending` |
| AC2 | `CreateMatchAsync` auto-inserisce una `MatchConfirmation(action=confirmed)` per il creator al momento della creazione | 1/4 già alla POST /match |
| AC3 | Ogni `POST confirm` conta le confirmations: se total == 4 → `match.Status = "confirmed"` atomicamente | Transizione in-transaction |
| AC4 | `POST dispute` → `match.Status = "disputed"` immediatamente, indipendentemente dalle altre conferme | Anche se 3/4 avevano già confermato |
| AC5 | Confirm/dispute verificano che il caller sia in `Team1Player1/2Id` o `Team2Player1/2Id` | 403 se non è uno dei 4 |
| AC6 | Unique index su `(MatchId, UserId)` in `MatchConfirmation`. Se la riga esiste già con stessa action → 200 idempotente | Nessun doppio conteggio |

**Edge case e requisiti impliciti:**
- Confirm su partita già `confirmed` o `disputed` → 409 (partita non più in pending)
- Non-autenticato → 401 (già garantito dal gruppo `.RequireAuthorization()`)
- Il GET pending mostra tutte le partite pending del circolo dove l'utente è partecipante, anche quelle già confermate dall'utente stesso (user vede lo stato globale, non solo le sue azioni residue)
- Dispute di una partita già `confirmed` → 409
- `MatchConfirmation` ON DELETE CASCADE rispetto a `Match`

## Soluzione tecnica

**Approccio:** aggiungere l'entità `MatchConfirmation` (tabella di tracking conferme), tre nuovi endpoint REST nel gruppo `/circles/{cId}/matches`, e una nuova view Angular per visualizzare e gestire le partite pending. La transizione `confirmed` avviene in modo sincrono nella stessa `SaveChangesAsync` del confirm — nessun background worker necessario.

**Decisioni chiave:**

| Decisione | Scelta | Alternativa scartata |
|-----------|--------|----------------------|
| Un endpoint `/respond` vs due separati | Due endpoint distinti (`/confirm`, `/dispute`) | `/respond` con body `{action}` — meno esplicito nel routing, errori meno chiari |
| Dispute: immediata vs maggioranza | Immediata (1 dispute → stato `disputed`) | Maggioranza — PRD è chiaro: 1 dispute è sufficiente a congelare |
| Pending: filtrato per "non ancora agito" vs tutti | Tutti i pending dove l'utente è giocatore | Solo quelli dove manca ancora la sua azione — più complesso, meno trasparente sul progresso collettivo |

**Modello dati — nuova entità:**

```csharp
public class MatchConfirmation
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public Guid MatchId { get; set; }
    public Guid UserId { get; set; }
    public string Action { get; set; } = "confirmed"; // "confirmed" | "disputed"
    public DateTimeOffset CreatedAt { get; set; } = DateTimeOffset.UtcNow;

    public Match Match { get; set; } = null!;
    public User User { get; set; } = null!;
}
```

Unique index: `(MatchId, UserId)`.

**Contratti API:**

```
POST /circles/{circleId}/matches/{matchId}/confirm
  Auth: Bearer JWT
  Resp 200: { matchId, status, confirmationsCount }
  Resp 403: non partecipante
  Resp 409: match non in pending

POST /circles/{circleId}/matches/{matchId}/dispute
  Auth: Bearer JWT
  Resp 200: { matchId, status: "disputed" }
  Resp 403: non partecipante
  Resp 409: match non in pending

GET /circles/{circleId}/matches/pending
  Auth: Bearer JWT
  Resp 200: [ { id, winnerTeam, team1, team2, sets, createdAt, confirmationsCount, userHasConfirmed } ]
  Resp 403: non membro del circolo
```
