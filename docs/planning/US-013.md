## Analisi requisiti

**Mappa AC → comportamento concreto:**
AC1 (solo `owner_id` può forzare): il caller deve essere il proprietario del circolo a cui appartiene la partita. Edge: l'owner può essere anche uno dei 4 giocatori — valido, nessun conflitto. AC2 (solo `pending`): `confirmed` e `disputed` restituiscono 400 (azione amministrativa, non conflitto di stato). Tentativo su partita di altro circolo → 404 prima perché il join `match.CircleId == circleId` fallisce. AC3 (ELO identico a US-007): richiama `IRatingService.CalculateAndApplyAsync` — stessa logica, zero duplicazione. AC4 (audit trail): due campi nullable su `Match` (`ForceConfirmedById` + `ForceConfirmedAt`) — distinguono la conferma forzata dalla 4/4 senza schemi aggiuntivi. AC5 (giocatori vedono `confirmed`): il `GET /matches` già restituisce `status`; nessuna modifica alla query. AC6 (multi-tenancy): garantita dalla sequenza — carica circolo per `circleId`, controlla `circle.OwnerId == userId`, poi carica partita verificando `match.CircleId == circleId`.

**Requisiti impliciti:** la `CircleSummary` DTO da `GET /circles/me` non espone `ownerId` → il frontend non può derivare `isOwner`. Va aggiunto. Open question (nota obbligatoria?) → body opzionale ignorato per MVP; log silenzioso è sufficiente per l'audit trail minimo.

---

## Soluzione tecnica

**Approccio:** nuovo endpoint `POST /circles/{circleId}/matches/{matchId}/force-confirm` in `MatchEndpoints.cs`. Nessun nuovo servizio; riusa `IRatingService`. Due campi nullable su `Match` per audit trail + una migration. Frontend: `ownerId` aggiunto a `CircleSummary` e pulsante condizionale in `circle-match-history.component`.

**Decisioni chiave:**

| Decisione | Scelta | Alternativa scartata |
|---|---|---|
| Audit trail | Campi `ForceConfirmedById`/`At` su `Match` | Tabella `MatchForceLog` separata — overkill per MVP |
| Owner detection FE | `ownerId` in `CircleSummary` (già da `GET /circles/me`) + lookup nel component | Nuovo endpoint `GET /circles/{id}` — endpoint in più non giustificato |
| Body richiesta | Nessun body obbligatorio | `reason` obbligatorio — risolto quando open question risolta |
| Stato dopo forza | `confirmed` diretto, no `MatchConfirmation` fake | Fake confirmations — inquina la semantica della tabella |

**Contratto API:**
```
POST /circles/{circleId}/matches/{matchId}/force-confirm
Authorization: Bearer <token>

200: { "status": "confirmed", "forceConfirmedBy": "<userId>" }
400: { "error": "La partita non è in stato pending" }
403: { "error": "Solo il proprietario del circolo può eseguire questa azione" }
404: { "error": "Partita non trovata" }
```
