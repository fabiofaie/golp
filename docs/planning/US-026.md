## Analisi requisiti

La validazione del token deve avvenire **prima** della domanda "Hai già usato GOLP?" (AC6), ma oggi `JoinByTokenAsync` consuma il join al primo utilizzo — non esiste un endpoint di sola lettura per verificare un token senza effetti collaterali. Serve un nuovo endpoint anonimo di validazione.

L'infrastruttura di passaggio token è già in gran parte presente: `JoinCircleComponent` legge `?token=` dalla query, e sia `LoginComponent` che `RegisterComponent` leggono già `?inviteToken=` e chiamano `circleSvc.joinByToken(...)` dopo l'autenticazione riuscita (AC2-4 già coperti dal codice esistente, nessuna modifica lì necessaria). Il vero gap è nel componente di atterraggio: oggi mostra due link diretti (Registrati / Accedi) invece di una domanda esplicita sì/no.

Edge case: se il token scade tra la validazione iniziale e la chiamata finale a `joinByToken` dopo login/registrazione, l'errore è già gestito dal flusso esistente (stesso comportamento di oggi). Nessuna ambiguità critica residua — le Open questions della storia sono vuote.

## Soluzione tecnica

Aggiungere un endpoint `GET /circles/invite/{token}` anonimo che valida il `JoinCode` senza consumarlo, restituendo nome circolo se valido o 404 se non valido/scaduto. `JoinCircleComponent.ngOnInit` chiama questo endpoint per primo: se invalido, mostra l'errore esistente (nessuna domanda); se valido e utente già autenticato, mantiene il comportamento attuale (auto-join via `joinByToken`, invariato); se valido e utente non autenticato, mostra il nuovo step "Hai già usato l'app GOLP?" che sostituisce i due link attuali — risposta Sì naviga a `/login`, No naviga a `/register`, entrambi con `inviteToken` già in query (routing invariato).

Nessuna modifica al modello dati: si riusa `Circle.JoinCode` esistente.

| Componente | Modifica |
|---|---|
| `CircleEndpoints.cs` | nuovo `GetInviteInfoAsync`, mappato su `GET /circles/invite/{token}` (AllowAnonymous) |
| `circle.service.ts` | nuovo metodo `getInviteInfo(token)` |
| `join-circle.component.ts` | refactor `ngOnInit`: valida token, stato `hasUsedGolp: boolean \| null`, `circleName` |
| `join-circle.component.html` | nuovo step domanda sì/no prima dei link esistenti |

**Contratto API:**
```
GET /circles/invite/{token}  (anonymous)
200 { valid: true, circleName: string }
404 { valid: false }   // token non trovato/non valido
```

**Rischi:** basso — riusa meccanismi esistenti (JoinCode, joinByToken, passthrough inviteToken in login/register), nessuna migrazione DB, nessun cambio di contratto sulle API esistenti.
