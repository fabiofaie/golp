## Analisi requisiti

**AC e comportamenti concreti:**
1. **Push ai 3 non-inseritori** — trigger: `POST /circles/{id}/matches` completa con 201 → backend individua i 4 player, esclude `CreatedById`, invia FCM ai restanti 3 tramite Firebase Admin SDK.
2. **Tap → schermata conferma** — il payload FCM deve includere `circleId` e `matchId`; il service worker Angular gestisce `notificationclick` aprendo `/circles/{circleId}/matches/{matchId}/confirm`.
3. **Permesso negato → app funziona** — token FCM non è prerequisito per nessuna funzionalità esistente; se l'utente nega il permesso, `getToken` lancia eccezione gestita silenziosamente; la partita è visibile nella lista pending via polling normale.
4. **Nessun duplicato** — upsert su `(UserId, Token)` in tabella `FcmTokens`; la chiamata `SendConfirmationRequestAsync` è fire-and-forget (non bloccante sulla creazione partita).

**Edge case identificati:**
- Utente con più device → push a tutti i token FCM registrati per quell'utente.
- Token FCM scaduto/invalido → Firebase restituisce `UNREGISTERED` → il token viene rimosso silenziosamente dal DB.
- FCM non raggiungibile → la partita è già persistita, la push fallisce silenziosamente (nessun rollback della partita).
- Inseritore escluso esplicitamente: filtro `recipientIds = allPlayerIds.Where(id => id != createdById)`.

**Requisiti impliciti:**
- VAPID key pubblica esposta via `GET /api/push/vapid-public-key` (necessaria ad Angular per `PushManager.subscribe`).
- Credenziali Firebase Admin (`serviceAccountKey.json`) mai committate — solo via `dotnet user-secrets` o env var.
- Service worker PWA: `ngsw-config.json` + `@angular/service-worker`; file `firebase-messaging-sw.js` nella root assets.

---

## Soluzione tecnica

**Approccio:** PWA layer su Angular 19 (`@angular/service-worker` + `@angular/fire/messaging`) + micro-layer notifiche sul backend .NET 10 (Firebase Admin SDK NuGet). Token FCM registrato dopo login, conservato in tabella `FcmTokens`. All'atto di `CreateMatch`, backend calcola i 3 destinatari e invia in background (fire-and-forget, `Task.WhenAll`).

**Componenti toccati:**

| Layer | Componente | Modifica |
|-------|-----------|----------|
| Backend | `FcmToken.cs` | **new** entity: `Id, UserId, Token, DeviceId, CreatedAt` |
| Backend | `AppDbContext.cs` | aggiunge `DbSet<FcmToken>` |
| Backend | `IPushNotificationService.cs` | **new** interfaccia |
| Backend | `PushNotificationService.cs` | **new** implementazione FirebaseAdmin |
| Backend | `PushEndpoints.cs` | **new** `POST /api/push/token`, `DELETE /api/push/token`, `GET /api/push/vapid-public-key` |
| Backend | `MatchEndpoints.cs` | dopo `SaveChangesAsync` → fire-and-forget `SendConfirmationRequestAsync` |
| Backend | `Program.cs` | registra Firebase Admin + `IPushNotificationService` |
| Backend | EF Migration | `AddFcmTokens` |
| Frontend | `angular.json` | `serviceWorker: true`, punta a `ngsw-config.json` |
| Frontend | `ngsw-config.json` | **new** config caching PWA |
| Frontend | `app.config.ts` | aggiunge `provideServiceWorker`, `provideFirebaseApp`, `provideMessaging` |
| Frontend | `environment.ts` | `firebaseConfig` (projectId, messagingSenderId, appId, vapidKey) |
| Frontend | `push-notification.service.ts` | **new** register/unregister token |
| Frontend | `firebase-messaging-sw.js` | **new** gestisce `notificationclick` → deep-link |
| Frontend | `auth.service.ts` | dopo login/register → `register()`; dopo logout → `unregister()` |

**Decisioni chiave:**
- **FirebaseAdmin NuGet** (`FirebaseAdmin` by Google) — ufficiale, mature, supporta `.NET 10`.
- **Fire-and-forget** per FCM in `CreateMatch` — la creazione partita non aspetta FCM; se FCM fallisce, la partita è già salvata. Tradeoff: nessuna retry automatica per MVP.
- **Token rimosso on UNREGISTERED** — pulizia proattiva evita accumulo di token morti.
- **`@angular/fire` v19** — compatibile con Angular 19; verificare release notes al momento dell'implementazione.

**Contratti API:**

```
POST /api/push/token
Authorization: Bearer <jwt>
Body: { "token": "string", "deviceId": "string" }
→ 204 NoContent

DELETE /api/push/token
Authorization: Bearer <jwt>
Body: { "token": "string" }
→ 204 NoContent

GET /api/push/vapid-public-key
→ 200 { "key": "BAxxx..." }
```

**Rischi:**
- Service Worker non funziona su `ng serve` (HTTP non-localhost) — documentare che il test push richiede `ng build` + `http-server` locale o `ng serve --ssl`.
- Credenziali Firebase in dev → `dotnet user-secrets`; in CI/prod → env var `Firebase__ServiceAccountJson`.
