## Analisi requisiti

**AC e comportamenti concreti:**
1. **Delta visibile dopo conferma** — `GET /circles/{circleId}/matches` restituisce lista partite, ognuna con `myDelta: int | null`. Null per pending/disputed; valore int per confirmed.
2. **Delta = valore applicato al rating** — `myDelta` è letto direttamente dal campo `Match.DeltaTeamXPlayerY` (salvato da US-007); non ricalcolato a runtime.
3. **Nessun elemento UI espone formula/expected score/rating avversari** — backend restituisce solo `myDelta` per il current user; nessun campo di calcolo intermedio (no `expectedScore`, no `teamRating`, no delta degli altri player).
4. **Pending/disputed → nessun delta** — `myDelta = null`; frontend non mostra nulla (non "0", non "−", assente).

**Edge case:**
- Utente non partecipante → `myDelta = null` (userId non matcha nessuna delle 4 posizioni player).
- Match confirmed ma US-007 non ancora DONE (campi delta = null nel DB) → `myDelta = null` → stessa UI di pending. Stato transitorio accettabile.
- Delta negativo: `-N pt` in rosso; positivo: `+N pt` in verde; zero: `+0 pt` in neutro.

**Requisiti impliciti:**
- Nuovo endpoint `GET /circles/{circleId}/matches` — non esiste (solo POST).
- La risposta include player names inline per evitare N+1 lookup frontend.
- Route Angular `/circles/:circleId/matches` nuova; link "Partite" da MyCircles.
- Lo stesso endpoint sarà riutilizzato da US-005 (partite pending da confermare) — la lista è completa, filtraggio per status gestito lato frontend/query-param future.

---

## Soluzione tecnica

**Approccio:** Aggiunge `GetMatchesAsync` a `MatchEndpoints`. Backend calcola `myDelta` mappando il `userId` JWT alle 4 posizioni player. Frontend: `CircleMatchHistoryComponent` alla route `/circles/:circleId/matches`; confirmed + delta → badge colorato; pending/disputed → badge stato.

**Componenti toccati:**

| Layer | Componente | Modifica |
|-------|-----------|----------|
| Backend | `MatchEndpoints.cs` | aggiunge `GetMatchesAsync` handler + `MapGet("/")` |
| Frontend | `match.service.ts` | aggiunge `PlayerInfo`, `MatchSummary`; metodo `getMatches(circleId)` |
| Frontend | `circle-match-history/` | **new component** `CircleMatchHistoryComponent` (ts + html + scss) |
| Frontend | `app.routes.ts` | aggiunge route `/circles/:circleId/matches` |
| Frontend | `my-circles.component.html` | aggiunge link "Partite" su ogni circle card |

**Contratto API:**

```
GET /circles/{circleId}/matches
Authorization: Bearer <jwt>
→ 200 [
    {
      "id": "guid",
      "status": "confirmed|pending|disputed",
      "winnerTeam": 1,
      "createdAt": "2026-06-12T...",
      "myDelta": 12,               // null se pending/disputed, null se utente non partecipante
      "confirmationsCount": 2,     // aggiunto per US-005 (progress 4-dot)
      "hasCurrentUserConfirmed": true, // aggiunto per US-005 (mostra/nasconde bottoni)
      "team1": [{ "userId": "...", "name": "..." }, { "userId": "...", "name": "..." }],
      "team2": [{ "userId": "...", "name": "..." }, { "userId": "...", "name": "..." }]
    }
  ]
→ 404  circolo non trovato
→ 401  non autenticato
```

**Logica myDelta backend:**
```csharp
int? myDelta = null;
if (match.Status == "confirmed")
{
    if      (match.Team1Player1Id == userId) myDelta = match.DeltaTeam1Player1;
    else if (match.Team1Player2Id == userId) myDelta = match.DeltaTeam1Player2;
    else if (match.Team2Player1Id == userId) myDelta = match.DeltaTeam2Player1;
    else if (match.Team2Player2Id == userId) myDelta = match.DeltaTeam2Player2;
}
```

**Decisioni chiave:**
- **`myDelta` personalizzato in backend** — ogni utente vede solo il proprio delta; coerente con PRD "algoritmo opaco".
- **Lista completa senza filtro status** — US-005 riutilizza lo stesso endpoint per pending. Filtro status gestito dal frontend.
- **`confirmationsCount` + `hasCurrentUserConfirmed`** — campi aggiunti per US-005 (4-dot progress e bottoni confirm/dispute). La query EF include count di MatchConfirmations e check per current user.
- **Player names inline** — la query EF include `User.Name` per i 4 player via join per evitare chiamate aggiuntive frontend.

**Rischi:**
- US-007 non DONE: `DeltaTeamXPlayerY` null → `myDelta` null su tutte le confirmed. Frontend deve gestire senza errori.
- US-005 non pianificata: nessuna partita sarà `confirmed` in produzione → la lista mostrerà solo pending.
