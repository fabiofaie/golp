## Analisi requisiti

US-018 introduce un endpoint owner-only per aggiungere un giocatore al circolo, con due rami: utente esistente (richiede conferma esplicita prima di aggiungere) e utente nuovo (creazione account "in attesa di attivazione" + invio email con istruzioni per impostare la password).

Edge case e requisiti impliciti: `User` (`src/Golp.Api/Data/Entities/User.cs`) oggi non ha un concetto di "pending activation". Si usa `PasswordHash = string.Empty` come marcatore implicito — nessuna migration necessaria. Va verificato che `BCrypt.Net.BCrypt.Verify(password, "")` ritorni `false` (o lanci eccezione gestita) per garantire che l'utente pending non possa loggare prima di impostare la password.

L'autorizzazione owner-only segue il pattern già usato in `CircleEndpoints.cs:324` (`circle.OwnerId != userId`). Il flusso di set-password riusa interamente `PasswordResetService` (`GenerateTokenAsync` + entità `PasswordResetToken`, scadenza 1h) — generato a creazione invece che su richiesta utente, stesso meccanismo del reset password esistente.

Validazione: email duplicata nello stesso circolo → messaggio informativo (non errore), come da AC. Formato email: stessa regex/validazione usata in `RegisterAsync` (`AuthEndpoints.cs`).

Assunzioni su Open questions del backlog (non bloccanti):
- Email in dev passa da `DevelopmentEmailService` (console-only), come per password reset — sufficiente per questa storia.
- Nome del nuovo utente è sempre richiesto dal form quando l'email non esiste già (l'AC lo impone esplicitamente), quindi mai vuoto alla creazione.

## Soluzione tecnica

Riuso totale dello schema `PasswordResetToken` per il link di attivazione iniziale, evitando una nuova entità o migration. Nuovo endpoint `POST /circles/{circleId}/members`, owner-only, due path di risposta in base all'esistenza dell'email.

**Approccio:**
1. Owner invia `{ email }` → se email esiste, API ritorna `{ exists: true, name }` senza side-effect.
2. Owner confer­ma con `{ email, confirmed: true }` → crea `CircleMembership` (rating 1000) se non già membro, invia email di notifica.
3. Se email non esiste, owner invia `{ email, name }` direttamente → crea `User` pending (`PasswordHash=""`) + `CircleMembership` + token attivazione + email con link set-password.

| Componente | Modifica |
|---|---|
| `CircleEndpoints.cs` | Nuovo endpoint `AddMemberAsync`, record `AddMemberRequest`/`AddMemberResponse` |
| `IEmailService` / `DevelopmentEmailService` | Nuovi metodi: `SendCircleActivationEmailAsync(email, activationLink)`, `SendAddedToCircleNotificationAsync(email, circleName)` |
| `PasswordResetService` | Nessuna modifica — riuso diretto di `GenerateTokenAsync(userId)` |
| DB | Nessuna migration: pending-state = `PasswordHash` vuoto |
| Frontend | Componente form in pagina gestione circolo, `circle.service.ts` nuovi metodi |

**Decisioni chiave:**
- Niente nuovo campo `IsActivated`: riduce superficie di modifica, ma rende lo stato "pending" implicito — commentare nel codice perché `PasswordHash==""` ha questo significato speciale.
- Endpoint singolo con flag `confirmed` invece di due endpoint separati: riduce duplicazione autorizzazione/validazione email, il frontend gestisce i due step come stati locali del form.

**Rischi:**
- `BCrypt.Verify` su hash vuoto: comportamento da verificare con test esplicito (TASK-9) prima di considerare sicuro il blocco login per utenti pending.
- Riuso `PasswordResetService.GenerateTokenAsync` invalida automaticamente token precedenti dell'utente (`InvalidatePreviousTokensAsync`) — comportamento desiderato anche qui (idempotenza se l'owner ripete l'azione), nessuna modifica necessaria.
