## Analisi requisiti

La storia riorganizza la dashboard in quattro sezioni prioritarie, ma dipende concretamente da concetti non ancora esistenti nel codice: "circolo attivo" (AC2, AC3, AC4) e "richieste urgenti" (AC1) non hanno oggi una fonte dati. `GET /circles/me` fornisce già MyRating/MyRank/MemberCount per ogni circolo, quindi AC2 è tecnicamente copribile senza attendere US-066 — ma serve un criterio provvisorio di "circolo attivo", da sostituire quando US-066 introdurrà un selettore esplicito. AC3 (serie vittorie) e AC4 (ultime partite) richiedono match `confirmed` ordinati per data nel circolo attivo; AC7 (aggregazione single-call) è demandato a US-070, quindi per questa storia è accettabile comporre lato frontend più chiamate REST esistenti.

Edge case da coprire: utente senza circoli (fuori scope diretto, stato onboarding pieno è US-069, ma qui non deve rompersi), zero azioni urgenti (sezione nascosta, non vuota-visibile), circolo attivo con meno di 4 membri (AC6, messaggio esplicativo sul CTA), utente con circoli ma zero partite confirmed (serie=0, elenco vuoto). Requisito implicito: la sezione azioni urgenti deve rispettare multi-tenancy — il filtro è già sicuro server-side via `userId` (nessun leak possibile), ma il rendering deve mostrare `circleName`/`sport` per ogni item essendo cross-circolo, altrimenti l'utente non capisce a quale circolo appartiene una richiesta.

**Decisioni utente (bloccanti, risolte prima del piano architetturale):**
1. Criterio "circolo attivo" provvisorio: se l'utente ha un solo circolo usa quello; se ne ha più di uno, usa il più vecchio per data di iscrizione (`CircleMembership.JoinedAt`, campo già esistente — nessuna migrazione). Verrà sostituito da US-066.
2. Azioni urgenti (AC1): tutte le pending dell'utente in **tutti** i circoli, non solo il circolo attivo — coerente con la decisione già presa nella conversazione di design che le richieste urgenti restano sempre globali.

## Soluzione tecnica

### Approccio
La dashboard passa da menu statico a componente con dati reali, componendo lato frontend le chiamate già esistenti `GET /circles/me` e `GET /my-matches?status=pending`. Il "circolo attivo" è calcolato client-side applicando il criterio provvisorio sui dati di `circles/me`. Le azioni urgenti aggregano tutte le partite pending dell'utente cross-circolo, filtrate solo per `userId` (già garantito dal backend via claims). Nessuna modifica al modello dati; tutto il lavoro è nel componente Angular più una piccola estensione del DTO `/circles/me`.

### Decisioni chiave

1. **Circolo attivo calcolato client-side, non persistito** — `CircleMembership.JoinedAt` esiste già nell'entity; va solo esposto nel DTO di `/circles/me` se manca. Isolato in funzione pura `pickActiveCircle(circles)` per essere sostituito facilmente da US-066.
2. **Azioni urgenti = `GET /my-matches?status=pending` senza filtro circolo** — la query filtra già per `userId` lato server, quindi è multi-tenant safe by construction. Nessun filtro `circleId` va aggiunto (contraddirebbe AC1).
3. **Niente endpoint aggregato ora (composizione N-chiamate)** — comporre 2 chiamate REST in parallelo è accettabile per il volume dati atteso. L'aggregazione single-call è AC7, demandata a US-070; introdurla ora sarebbe scope creep.
4. **Nessun componente bottom-nav condiviso** — il "+" resta un CTA inline nel template della dashboard. Costruire uno shell bottom-nav ora significherebbe indovinare l'API di US-064 senza i suoi AC.
5. **Guardia AC6 è puro if lato frontend** — `memberCount` è già in `circles/me`, nessuna nuova chiamata.

### Modifiche al modello dati

Nessuna. `CircleMembership.JoinedAt` esiste già (`DateTime`, default `UtcNow`).

### Contratti API

Nessun nuovo endpoint. Riuso:
- `GET /circles/me` → aggiungere `joinedAt` al DTO se manca.
- `GET /my-matches?status=pending` → usato as-is, nessun filtro `circleId`.

### Rischi

- **Debito tecnico N-chiamate** → segnalare come input per US-070 quando verrà pianificata.
- **Criterio circolo attivo provvisorio** → isolato in funzione pura per rendere la sostituzione futura (US-066) un cambio puntuale.
- **UX su azioni urgenti cross-circolo** → ogni item deve mostrare `circleName`/`sport`, altrimenti l'utente perde il contesto.

**Out of scope architetturale:** endpoint aggregato (US-070), selettore circolo persistito (US-066), bottom-nav completa condizionale (US-064).
