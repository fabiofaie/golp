## Analisi requisiti

Gli AC della storia mappano su due filoni distinti che condividono solo il tema "Game+Bonus": (1) una pagina di spiegazione+simulatore gemella di `/elo-info`, e (2) rendere "coerente" il link a tale pagina in base al `RatingMethod` del circolo. Per il filone (2), l'AC "ovunque nell'app compaia oggi il link... punta invece a..." va letto con un vincolo implicito: **serve un circolo di riferimento per sapere quale metodo applicare**. `circle-leaderboard.component.ts` ha già `ratingMethod` in memoria (riga 25/48) — lì il link diventa banalmente condizionale. La dashboard (`dashboard.component.ts`) invece non è legata a nessun circolo specifico (è l'hub globale post-login): non ha un `RatingMethod` da cui decidere. Risolvo l'ambiguità così: la card "Simula una partita" in dashboard resta un link generico a `/elo-info` (comportamento invariato, nessun circolo di riferimento disponibile), mentre ogni punto dell'app che *è* già dentro il contesto di un circolo (leaderboard) usa il metodo di quel circolo. Questo non contraddice l'AC ("ovunque compaia... per un dato circolo") perché la dashboard non è "per un dato circolo".

Riuso di formula: `GameBonusRatingService.CalculateAndApplyAsync` oggi accoppia il calcolo dei punti (puro) al fetch EF della finestra rolling (DB-dependent). Per il simulatore serve solo la parte pura (base + bonus upset a partire da input utente), quindi va estratto un metodo statico puro `ComputeMatchPoints(winnerUnits, loserUnits, winnerCurrentScore, loserCurrentScore)` riusato sia dal servizio reale sia dal nuovo endpoint di simulazione — stesso pattern già usato da US-017 con `RatingService.ComputeDeltas`. Nessuna ambiguità critica bloccante: le 2 individuate sopra sono risolte qui stesso (dashboard generica; riuso formula via extract method).

Edge case da coprire nel simulatore Game+Bonus: punteggi correnti a 0 per entrambe le squadre (nessun bonus possibile, come da US-052 "Open questions" già risolta); parità di punteggio corrente (bonus = 0, non negativo); game vinti = game persi non è un input valido (una squadra deve vincere, come nel simulatore ELO che già impone `team1Won = totalTeam1 >= totalTeam2` — qui invece serve un vincitore esplicito perché a game pari non esiste "vincitore" nel Game+Bonus reale, quindi valido solo se i game totali differiscono, o se il form fa scegliere esplicitamente la squadra vincente per gli sport senza set/pareggio).

## Soluzione tecnica

Approccio: duplicare la struttura di `/elo-info` in una nuova route `/game-bonus-info` (component + service dedicati, niente componente condiviso parametrizzato — i due metodi hanno testi e form differenti, forzare un'astrazione comune ora costerebbe più di quanto risparmia). Backend: nuovo endpoint pubblico `POST /api/simulate-game-bonus` che riusa la logica pura estratta da `GameBonusRatingService`. La "coerenza del link" si risolve con un helper di puro routing lato frontend (funzione `ratingInfoPath(method)` in un file condiviso, es. `circles/rating-method.util.ts`) usato da `circle-leaderboard` per scegliere fra `/elo-info` e `/game-bonus-info`; la dashboard non cambia.

**Decisioni chiave:**
- **Pagina duplicata, non parametrizzata**: `EloInfoComponent` e nuovo `GameBonusInfoComponent` restano indipendenti. Tradeoff: un po' di duplicazione di markup/stile (accettabile, sono ~200 righe di HTML statico) contro evitare un componente con troppi `@if` per due domini concettualmente diversi (rating assoluto vs punti finestra).
- **Endpoint separato `/api/simulate-game-bonus`**: non riuso `/api/simulate-match` con un flag `method`, perché richiede input radicalmente diversi (rating vs punteggi correnti + game). Coerente con come US-017 ha già isolato `/api/simulate-match` come endpoint pubblico dedicato.
- **Estrazione di `GameBonusRatingService.ComputeMatchPoints` (static, puro)**: elimina la duplicazione della formula base+bonus tra il calcolo reale (DB) e la simulazione (nessun DB). Il metodo esistente `CalculateAndApplyAsync` viene refattorizzato per chiamare il nuovo metodo statico invece di reimplementare la formula inline.
- **Simulatore richiede input esplicito di chi vince** (non derivato solo dai game) per gli sport senza set/pareggio ammesso a game pari: evita un edge case ambiguo nel form senza introdurre logica speciale per sport.

**Contratto API nuovo endpoint:**
```
POST /api/simulate-game-bonus
{
  "myGames": 6, "oppGames": 3,      // o "sets": [...] per sport a set — stesso pattern di SimulateMatchRequest
  "team1CurrentScore": 42, "team2CurrentScore": 30,
  "team1Won": true                   // esplicito, vedi edge case sopra
}
→ 200 { "team1Points": 4, "team2Points": 0 }
```
Validazione: game/punteggi correnti non negativi, almeno un punteggio > 0 (stesso pattern di `SimulateEndpoints`), nessuna autenticazione richiesta (pubblico, come `/api/simulate-match`).

**Rischi:**
- Duplicazione di logica leaderboard→link se in futuro compaiono altri punti "per circolo" con link rating: mitigato dall'helper condiviso `rating-method.util.ts`, quindi il rischio è basso e contenuto.
- La formula Game+Bonus reale include set (somma game su tutti i set, US-012 pattern): il simulatore deve replicare lo stesso criterio di somma usato da `MatchEndpoints`/`GameBonusRatingService` (`match.Sets.Sum(...)`) — rischio di disallineamento se lo sport ha `sets:true` e il form permette solo un risultato unico. Task dedicato per gestire risultato-per-set nel form, analogo a `elo-info` (`resultMode: 'unico' | 'set'`).
