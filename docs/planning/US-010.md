## Analisi requisiti

**AC e comportamenti concreti:**
1. **Giocatore del mese** — filtra partite `confirmed` del circolo con `CreatedAt` nel mese/anno corrente → per ogni giocatore somma il proprio delta (positivo + negativo) → vincitore = massimo net gain. Winner null se nessuna partita confirmed nel periodo.
2. **Giocatore dell'anno** — identica logica, filtro su anno corrente (1 gen → 31 dic).
3. **Reset al cambio mese/anno** — il "reset" avviene naturalmente filtrando per data; nessuna colonna di stato da azzerare manualmente.
4. **Storico consultabile** — premi passati salvati in tabella `CircleAward` con lazy population: primo accesso a un periodo chiuso → calcola e memorizza.
5. **Nessun premiato → no errori** — `winner: null` nella risposta; UI mostra stato vuoto chiaro.
6. **Per-circolo** — tutte le query filtrate su `CircleId`; mai cross-circolo.

**Edge case:**
- Parità di net gain → tie-break: `matchesPlayed DESC`, poi `UserId ASC` (deterministico).
- Giocatore con solo sconfitte nel periodo (net gain negativo) → può vincere se è l'unico con partite confirmed.
- Delta null (US-007 non ancora DONE) → trattato come 0; partita confirmed inclusa ma contribuisce 0 al net gain.
- Periodo corrente (ongoing): SEMPRE ricalcolato fresh, mai cachato in `CircleAward`.

**Requisiti impliciti:**
- Entità `CircleAward` per storico: `Id, CircleId, WinnerUserId (nullable), PeriodType, PeriodYear, PeriodMonth (nullable), TotalDelta (nullable), ComputedAt`.
- Endpoint unico `GET /circles/{id}/awards` per MVP → `{ currentMonth, currentYear }`.
- Route `/circles/:circleId/awards` nuova; link "Premi" da MyCircles.

---

## Soluzione tecnica

**Approccio:** Nuovo endpoint `GET /circles/{id}/awards` aggrega delta direttamente da `Matches` filtrati per `CircleId + Status=confirmed + CreatedAt in range`. Calcolo on-the-fly per periodo corrente; lazy cache in `CircleAward` per periodi passati. Nessun job schedulato necessario per MVP.

**Componenti toccati:**

| Layer | Componente | Modifica |
|-------|-----------|----------|
| Backend | `CircleAward.cs` | **new** entity |
| Backend | `AppDbContext.cs` | aggiunge `DbSet<CircleAward>` |
| Backend | EF Migration | `AddCircleAwards` |
| Backend | `AwardsEndpoints.cs` | **new**: `GET /circles/{id}/awards` |
| Backend | `Program.cs` | registra `MapAwardsEndpoints` |
| Frontend | `circle.service.ts` | aggiunge `AwardWinner`, `AwardPeriodResult`, `CircleAwardsResponse`, `getAwards()` |
| Frontend | `circle-awards/` | **new component** `CircleAwardsComponent` (ts + html + scss) |
| Frontend | `app.routes.ts` | aggiunge route `/circles/:circleId/awards` |
| Frontend | `my-circles.component.html` | aggiunge link "Premi" su ogni card |

**Query aggregazione (schema LINQ in-memory dopo fetch):**
```csharp
// Recupera partite confirmed nel periodo dal DB
var matches = await db.Matches
    .Where(m => m.CircleId == id && m.Status == "confirmed" && m.CreatedAt >= periodStart && m.CreatedAt < periodEnd)
    .ToListAsync();

// Aggrega in-memory (SelectMany su 4 player-delta pairs)
var playerDeltas = matches
    .SelectMany(m => new[] {
        (UserId: m.Team1Player1Id, Delta: m.DeltaTeam1Player1 ?? 0),
        (UserId: m.Team1Player2Id, Delta: m.DeltaTeam1Player2 ?? 0),
        (UserId: m.Team2Player1Id, Delta: m.DeltaTeam2Player1 ?? 0),
        (UserId: m.Team2Player2Id, Delta: m.DeltaTeam2Player2 ?? 0),
    })
    .GroupBy(x => x.UserId)
    .Select(g => new { UserId = g.Key, NetGain = g.Sum(x => x.Delta), Matches = g.Count() })
    .OrderByDescending(x => x.NetGain).ThenByDescending(x => x.Matches).ThenBy(x => x.UserId)
    .FirstOrDefault();
```

**Contratto API:**
```
GET /circles/{id}/awards
Authorization: Bearer <jwt>
→ 200 {
    "currentMonth": {
      "period": "2026-06",
      "winner": { "userId": "...", "name": "...", "netGain": 42, "matchesPlayed": 5 } | null
    },
    "currentYear": {
      "period": "2026",
      "winner": { "userId": "...", "name": "...", "netGain": 150, "matchesPlayed": 20 } | null
    }
  }
→ 404  circolo non trovato
→ 401  non autenticato
```

**Decisioni chiave:**
- **On-the-fly per periodo corrente** — evita scheduled job; query leggera per MVP (poche partite per circolo amatoriale).
- **Lazy cache per periodi passati** — `CircleAward` popolata al primo accesso di un periodo chiuso; semplice da implementare, no race condition critica per MVP.
- **Delta null → 0** — `?? 0` include partite confirmed senza delta (US-007 non ancora DONE) senza errori.
- **Aggregazione in-memory** — con dataset piccolo (circolo amatoriale ≤ 30 partite/mese) l'aggregazione LINQ in-memory è più leggibile del GroupBy SQL su 4 colonne.

**Rischi:**
- US-007 non DONE: tutti i delta null → net gain 0 per tutti → nessun premiato con dati significativi.
- Lazy cache senza locking: richieste concorrenti al primo accesso di un periodo passato possono creare 2 record; mitigato con indice unico su `(CircleId, PeriodType, PeriodYear, PeriodMonth)` che causa eccezione gestita.
