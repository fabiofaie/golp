## Analisi requisiti

**AC mapping concreto:**

- **AC1** — 4 giocatori distinti, tutti membri del circolo, in 2 coppie. Body: `team1: [id, id]`, `team2: [id, id]`. Backend verifica no duplicati tra i 4 ID (set di 4 unici) e membership attiva in `CircleMemberships` per ogni ID nel circolo corrente.
- **AC2** — Punteggio coerente con `Circle.Sets`: se `Sets=true` → array `sets` con ≥1 elemento `{team1, team2}`; se `Sets=false` → array con esattamente 1 elemento. Vincitore = team con più set vinti (`Sets=true`) o score maggiore (`Sets=false`).
- **AC3** — `userId` da JWT deve comparire in `team1` o `team2`. Nessun campo nel body — si ricava dal token.
- **AC4** — Duplicato tra i 4 IDs → 400. Non membro → 400.
- **AC5** — Set vinti pari (`Sets=true`) o score uguale (`Sets=false`) → 400. La partita deve avere un vincitore determinabile.
- **AC6** — `Status = "pending"` al create. Nessuna logica ELO qui — toccata da US-006 dopo `confirmed`.

**Open question risolta:** punteggio per set accettato libero (nessuna regola sport-specifica come 6-4 padel). Validazione solo su presenza sets e vincitore determinabile. PRD usa `score_ratio = punti_vince / totale_punti` — indipendente dalla scala.

**Requisiti impliciti:** circolo non trovato → 404; utente autenticato ma non membro del circolo → 403; `CreatedById` ricavato dal JWT; `CreatedAt = UtcNow`.

## Soluzione tecnica

**Approccio:** due nuove entità EF Core (`Match`, `MatchSet`), un nuovo endpoint `POST /circles/{id}/matches`, un nuovo componente Angular `RecordMatchComponent`. Zero modifiche allo schema esistente — solo aggiunte.

**Decisioni chiave:**

| Decisione | Scelta | Alternativa scartata |
|---|---|---|
| Score storage | Tabella `MatchSets` separata (normalizzata) | JSON su `Match` — meno queryable per US-006 ELO |
| `WinnerTeam` salvato al create | Sì (int: 1 o 2) su `Match` | Ricalcolo ogni query — peggio per US-006 |
| Validazione score | Libera — solo winner check | Rule-based per sport — over-engineering MVP |
| Route Angular | `circles/:id/match/new` (nested) | `/record-match/:id` — meno intuitivo |

**Modello dati:**

```
Match:
  Id (Guid, PK)
  CircleId (Guid, FK → Circle)
  CreatedById (Guid, FK → User)
  Status (string: "pending")
  WinnerTeam (int: 1|2)
  Team1Player1Id, Team1Player2Id (Guid, FK → User)
  Team2Player1Id, Team2Player2Id (Guid, FK → User)
  CreatedAt (DateTimeOffset)

MatchSet:
  Id (Guid, PK)
  MatchId (Guid, FK → Match, cascade delete)
  SetNumber (int)
  Team1Score (int)
  Team2Score (int)
```

**Contratto API:**

```
POST /circles/{circleId}/matches
Authorization: Bearer
Body: {
  team1: [guid, guid],
  team2: [guid, guid],
  sets: [{ team1: int, team2: int }, ...]
}
→ 201: { id, circleId, status: "pending", winnerTeam: 1|2, createdAt }
→ 400: { error: "..." }   // duplicato, non membro, tie, sets invalidi
→ 403: { error: "Non sei membro del circolo" }
→ 404: { error: "Circolo non trovato" }
```
