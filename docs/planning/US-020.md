## Analisi requisiti

AC1-3 (template engine + layout + migrazione 3 email esistenti) sono refactor isolato e ben definito — `SmtpEmailService.cs` ha già la firma `SendAsync(toEmail, subject, htmlBody)` come unico punto di invio, quindi inserire un renderer di template tra "scelta evento" e "costruzione HTML" è chirurgico. AC4 (email su creazione partita) si aggancia allo stesso pattern fire-and-forget già usato per il push in `CreateMatchAsync` (`MatchEndpoints.cs:130-145`) — serve solo l'email dei 3 destinatari e link alla pagina di conferma (route frontend `/circles/:circleId/matches/:matchId`, già esistente da US-005). AC5 (email su dispute) richiede l'email del *proprietario* del circolo, non presente nello scope attuale di `DisputeMatchAsync` (legge solo `match`, non `circle.Owner`) — serve query aggiuntiva. AC6 (non bloccante) impone fire-and-forget anche per l'email, stesso pattern try/catch+log del push. AC7 richiede che anche `DevelopmentEmailService` usi i template (stessa interfaccia, render uguale, solo l'output cambia da SMTP a console).

Edge case impliciti: nessuna ambiguità critica oltre quanto già risolto in AN-001 (push+email complementari, no opt-out, scope separato per i premi mensili).

## Soluzione tecnica

Introduco `IEmailTemplateRenderer` che carica file `.html` da `EmailTemplates/` (copiati in output a build time), fa sostituzione placeholder con `string.Replace` (niente regex/engine — placeholder pochi e noti), e compone layout+contenuto. `SmtpEmailService`/`DevelopmentEmailService` non costruiscono più HTML inline: chiamano il renderer con nome template + dictionary di valori, poi passano il risultato a `SendAsync` (SMTP) o console (dev).

| Componente | Modifica |
|---|---|
| `Services/IEmailTemplateRenderer.cs` (nuovo) | `string Render(string templateName, Dictionary<string,string> values)` |
| `Services/EmailTemplateRenderer.cs` (nuovo) | Carica `_layout.html` + `{templateName}.html` da `EmailTemplates/`, sostituisce `{{Placeholder}}` |
| `EmailTemplates/_layout.html`, `password-reset.html`, `circle-activation.html`, `added-to-circle.html`, `match-confirmation-request.html`, `match-disputed.html` (nuovi file) | Markup HTML con placeholder |
| `Golp.Api.csproj` | `<Content Include="EmailTemplates\**\*.html" CopyToOutputDirectory="PreserveNewest" />` |
| `IEmailService` | Resta con metodi semantici esistenti + 2 nuovi: `SendMatchConfirmationRequestEmailAsync`, `SendMatchDisputedEmailAsync` |
| `SmtpEmailService.cs`, `DevelopmentEmailService.cs` | Ogni metodo chiama `_renderer.Render(...)` invece di costruire stringa inline |
| `Endpoints/MatchEndpoints.cs` | `CreateMatchAsync`: nello stesso blocco fire-and-forget del push, aggiunge invio email ai 3 `recipientIds`; `DisputeMatchAsync`: dopo `SaveChangesAsync`, fire-and-forget email a `circle.Owner` (query aggiuntiva) |

**Decisioni chiave:**
- Niente template engine esterno (Razor/Scriban): placeholder semplici, `string.Replace` basta — coerente con AN-001.
- Layout condiviso via file `_layout.html` con marker `{{Body}}`: ogni template specifico fornisce solo il contenuto centrale, branding centralizzato in un solo posto.
- Email su dispute fa una query aggiuntiva per il circolo (oggi `DisputeMatchAsync` non lo carica) — costo minimo, nessun impatto su AC di storie precedenti (US-013/US-014).
- Riuso del pattern fire-and-forget esistente (`IServiceScopeFactory`+`Task.Run`+try/catch+log) invece di crearne uno nuovo: consistenza con il codice già in produzione per il push.

**Rischi:**
- File template `.html` devono essere copiati in `bin/` a build time (`CopyToOutputDirectory`) — se non funziona nel deploy Azure App Service (pipeline diversa dal build locale), il renderer fallisce in produzione. Da verificare con un deploy di test.
- Fire-and-forget per email di dispute introduce una seconda query DB dopo la response già inviata — stesso pattern già accettato in produzione per il push, nessun rischio nuovo.
