## Analisi requisiti

### Mapping AC → comportamento concreto

Il flusso Quick Match è un componente Angular multi-step con stato interno. Entry point: pulsante "Registra Partita" in `dashboard.component.html` → `/match/quick`. Il flusso classico (`circles/:circleId/match/new`) resta invariato.

Il **check circolo** usa due modalità distinte che dipendono dalla presenza di ospiti nuovi (non in DB):

**Modalità EXACT** — tutti i 4 slot risolti (nessun nuovo ospite):
- Il check invia `{sport, userIds, guests:[{email?,phone?}]}`. Il backend tenta di risolvere i guest per email/telefono. Se tutti e 4 risolti → cerca circoli con `MemberCount == 4` e tutti e 4 presenti.
- `length === 1` → redirect obbligatorio, nessun bypass
- `length > 1` → picker
- `length === 0` → crea nuovo

**Modalità PARTIAL** — almeno un guest non risolvibile (persona mai in DB):
- Il backend cerca circoli dove **tutti i player noti** (userId risolti) sono membri, stesso sport. Non impone `MemberCount == 4` e non richiede che il guest (sconosciuto) sia membro.
- Se trova circoli → picker CON opzione "Crea nuovo gruppo" sempre presente (l'utente può scegliere di registrare nel circolo esistente aggiungendo il nuovo ospite, o creare un nuovo gruppo)
- Se non trova circoli → crea nuovo direttamente

Il **check endpoint** risponde con `{circles, mode: 'exact'|'partial'}` per permettere al frontend di mostrare la UX corretta.

In `POST /match/quick`, quando l'utente sceglie un circolo esistente in modalità partial, il backend aggiunge il nuovo ghost user come `CircleMembership` del circolo scelto (già gestito da `ResolveOrCreateGuestAsync`).

### Edge case rilevanti

| Scenario | Modalità | Comportamento |
|----------|----------|---------------|
| 4 utenti registrati, circolo condiviso | EXACT | Redirect obbligatorio |
| 3 registrati + 1 ghost già in DB (email match) | EXACT (ghost risolto) | Cerca exact match con tutti e 4 |
| 3 registrati + 1 guest nuovo | PARTIAL | Picker con opzione "Crea nuovo" |
| 4 guest nuovi | PARTIAL (0 noti) | Nessun circolo trovato → crea nuovo |
| Utente senza storia né circoli | EXACT/PARTIAL | Suggestions vuota, solo search |

**Race condition:** ghost user creato tra check e submit → improbabile (flusso guidato, pochi secondi); se accade, il submit risolve correttamente.

### Requisiti impliciti
- Suggestions filtra solo dati visibili all'utente (multi-tenancy).
- La transazione atomica include `MatchConfirmationTokens`.
- US-039 REVIEW: `PlayerSlotDto` e `ResolveOrCreateGuestAsync` potrebbero cambiare — dependency da monitorare.
- `proxy.conf.json` non ha il prefisso `/match` → aggiunto in TASK-6.

## Soluzione tecnica

### Approccio
Tre endpoint backend in `QuickMatchEndpoints.cs`. Il check unificato (`POST /match/quick/check`) gestisce entrambe le modalità restituendo `mode` nel response. Il frontend rende la UX corretta in base al mode. Nessuna migrazione EF — `IsPrivate` esiste già su `Circle`.

### Contratti API

```
GET /match/quick/suggestions?sport={sport}&q={query}
Authorization: Bearer
→ 200 [{ userId, name, isActivated }]
  Max 20. Esclude utente corrente.
  Union: partite recenti + membri circoli, dedup per recency.

POST /match/quick/check
Authorization: Bearer
body: {
  sport: string,
  userIds: string[],                        // player registrati (risolti)
  guests: [{ email?: string, phone?: string }]  // ospiti (da risolvere)
}
→ 200 {
  mode: 'exact' | 'partial',
  circles: [{ id, name, lastMatchAt }]
}

  Logica backend:
  1. Per ogni guest: cerca User per email/phone → se trovato aggiunge a knownIds
  2. unresolvedCount = guest non trovati in DB
  3. if unresolvedCount === 0 && knownIds.Count === 4:
       mode = 'exact'
       → cerca Circle con MemberCount==4, tutti e 4 presenti, stesso sport
  4. else:
       mode = 'partial'
       → cerca Circle dove TUTTI i knownIds sono membri, stesso sport (no count constraint)

POST /match/quick
Authorization: Bearer
body: {
  sport: string,
  circleId?: string,       // se scelto (check o picker)
  circleName?: string,     // solo se circleId assente
  team1: PlayerSlotDto[],  // 2 slot
  team2: PlayerSlotDto[],  // 2 slot
  sets: SetScoreDto[]
}
→ 201 { circleId, matchId, circleName, circleCreated: bool }
```

### Logica check — query EF Core

```csharp
// EXACT (unresolvedCount == 0)
db.Circles
  .Where(c => c.Sport == sport
           && db.CircleMemberships.Count(m => m.CircleId == c.Id) == 4
           && allIds.All(pid =>
               db.CircleMemberships.Any(m => m.CircleId == c.Id && m.UserId == pid)))

// PARTIAL (unresolvedCount > 0)
db.Circles
  .Where(c => c.Sport == sport
           && knownIds.All(pid =>
               db.CircleMemberships.Any(m => m.CircleId == c.Id && m.UserId == pid)))
```

### Transazione atomica — POST /match/quick (nuovo circolo)

```
await using var tx = await db.Database.BeginTransactionAsync();
1. Crea Circle { IsPrivate=true, Name=circleName??AutoName(sport,slots) }
2. ResolveOrCreateGuestAsync per ogni slot guest (riuso da MatchEndpoints)
3. Aggiungi CircleMembership per tutti i 4 (rating 1000, idempotente)
4. Crea Match + MatchSets
5. Crea MatchConfirmationTokens
6. SaveChangesAsync() + CommitAsync()
Fire-and-forget: push + email
```

Quando `circleId` fornito (circolo esistente): step 1 e 3 saltati, `ResolveOrCreateGuestAsync` aggiunge il ghost al circolo scelto se non già membro.

### Rischi
- **US-039 REVIEW:** `ResolveOrCreateGuestAsync` potrebbe cambiare. Valutare estrazione in helper condiviso se le due storie si sovrappongono.
- **Partial mode ampio:** in partial, la query non vincola `MemberCount == 4` — potrebbe mostrare circoli con molti più membri in cui i 3 giocatori noti partecipano. Accettabile: l'utente sceglie consapevolmente.
