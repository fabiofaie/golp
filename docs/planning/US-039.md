## Analisi requisiti

La storia introduce due comportamenti distinti: (1) **guest resolution** — logica BE che, dato un contatto (email/phone), trova o crea un `User` e lo aggiunge al circolo; (2) **guest slot UI** — componente frontend che sostituisce il dropdown membro con campi nome+contatto + Contact Picker progressivo.

Il punto critico è `User.Email`: attualmente `IsRequired()` con unique index. Se l'ospite fornisce solo telefono, l'email deve diventare nullable. La soluzione corretta è rendere `Email` nullable con filtered unique index SQL Server (`WHERE Email IS NOT NULL`). L'alternativa (email placeholder fittizio) inquina i dati. La colonna `PasswordHash` resta `string.Empty` come già fa `AddMemberAsync` in `CircleEndpoints`.

Manca `User.Phone` (non esiste nell'entità) e `User.IsActivated` (attualmente il check ghost è implicito via `PasswordHash == ""`). `AddMemberAsync` in `CircleEndpoints` crea già ghost user — andrà allineato a `IsActivated = false`. `GetCircleMembersAsync` e leaderboard devono esporre `isActivated` per il badge "(non registrato)". La guest resolution deve avvenire **prima** della validazione "4 giocatori distinti" per controllare gli ID risolti.

---

## Soluzione tecnica

Estendere `User` con `Phone?` e `IsActivated`, rendere `Email` nullable. Cambiare il contratto di `CreateMatchAsync` da `Guid[]` a `PlayerSlotDto[]` (userId OR guest fields). La guest resolution avviene atomicamente in `CreateMatchAsync`: lookup → find-or-create User → find-or-create CircleMembership → match creation, tutto in un unico `SaveChangesAsync`.

### Modifiche modello dati

```csharp
// User.cs — aggiunte
public string? Email { get; set; }     // nullable (era string)
public string? Phone { get; set; }
public bool IsActivated { get; set; } = true;
```

```csharp
// AppDbContext — OnModelCreating
e.Property(u => u.Email).HasMaxLength(254);  // rimuovi .IsRequired()
e.HasIndex(u => u.Email).IsUnique().HasFilter("Email IS NOT NULL");
e.Property(u => u.Phone).HasMaxLength(30);
```

### Contratto API

```
// Nuovo DTO per ogni slot giocatore
PlayerSlotDto: {
  userId?: Guid,           // presente se slot = membro
  guestName?: string,      // presente se slot = ospite
  guestEmail?: string,     // almeno uno tra email/phone obbligatorio per ospite
  guestPhone?: string
}

CreateMatchRequest: {
  team1: PlayerSlotDto[2],
  team2: PlayerSlotDto[2],
  sets: SetScoreDto[]
}
```

Validazione slot: ogni slot deve avere o `userId` o (`guestName` + almeno uno tra `guestEmail`/`guestPhone`).

### Guest resolution (in CreateMatchAsync)

```
per ogni slot:
  if userId → verifica membro circolo (come oggi)
  else → lookup per email (normalizzata lowercase)
       → se non trovato, lookup per phone
       → se non trovato, crea new User { IsActivated=false, PasswordHash="" }
       → se non già membro, crea CircleMembership { Rating=1000 }
       → accumula userId risolto

poi → verifica 4 ID distinti → crea Match → SaveChangesAsync()
```

Tutta la guest resolution usa la stessa `db` instance (EF tracking) per atomicità — un unico `SaveChangesAsync` finale.
