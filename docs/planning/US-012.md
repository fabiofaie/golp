## Analisi requisiti

La formula corrente in `RatingService` somma tutti i game di tutti i set e ignora la struttura a set (`score_ratio = winnerUnits / totalUnits`). Per sport con `Circle.Sets = true` (padel, beachtennis) questo penalizza chi vince 2-0 di misura rispetto a chi vince 2-0 di netto, e sottovaluta una vittoria 2-1 rispetto alla percentuale di game vinti.

Campo `Circle.Sets` già presente nel modello — nessuna migrazione necessaria. `SportsConfig.SportDto` manca del parametro `SetWeight`. `RatingService` non carica la `Circle` entity (solo `Match.Include(Sets)`) e quindi oggi non sa se lo sport del circolo prevede set.

Edge case:
- **Circle non trovata in DB** (test unit attuali non la creano): fallback a formula game-only → nessun test esistente si rompe.
- **totalSets == 0** (tutti i set con score pari, es. 0-0): `set_ratio = 0.5`.
- **Sport senza sets** (`SetWeight = 0.0`): formula game-only identica a oggi.
- **Vincitore con meno game totali** (6-4, 0-6, 7-6): con formula blended `≈ 0.536 > 0.5` → delta = 1 invece di 0. Cambio di comportamento voluto: il 2-1 conta.

## Soluzione tecnica

Aggiungere `.Include(m => m.Circle)` alla query EF esistente in `RatingService.CalculateAndApplyAsync`. Leggere `match.Circle?.Sets` e `SportsConfig.GetBySport(match.Circle?.Sport)?.SetWeight` per scegliere tra formula blended e game-only. Aggiungere `double SetWeight = 0.0` a `SportsConfig.SportDto` con valore 0.4 per padel e beachtennis.

| Componente | Modifica |
|---|---|
| `SportsConfig.cs` | Nuovo campo `double SetWeight = 0.0` su `SportDto`; padel/beachtennis → 0.4 |
| `RatingService.cs` | `.Include(m => m.Circle)`; branch blended se `Circle.Sets && SetWeight > 0` |
| `RatingServiceTests.cs` | Fixture con opt-in `hasSets`; 4 nuovi test case |
| `RatingServiceIntegrationTests.cs` | Seed opzionale Circle entity; 1 nuovo test |

Nessuna modifica a `IRatingService`, nessuna migrazione, nessun impatto frontend.

Tradeoff: caricare `Circle` aggiunge 1 JOIN per conferma partita — nella stessa transazione di conferma, impatto trascurabile. L'alternativa (passare `setWeight` al metodo) romperebbe l'interfaccia `IRatingService` senza beneficio reale.

### TASK-2 — RatingService: formula blended ✅ (2026-06-15 09:00)
**File:** `src/Golp.Api/Services/RatingService.cs`

- Aggiungi `.Include(m => m.Circle)` alla query del Match (accanto a `.Include(m => m.Sets)`)
- Dopo il calcolo di `totalTeam1` / `totalTeam2`, determina se usare blended:
  ```csharp
  var sport = SportsConfig.GetBySport(match.Circle?.Sport ?? "");
  bool useBlended = (match.Circle?.Sets == true) && (sport?.SetWeight > 0);
  ```
- Se `useBlended`:
  ```csharp
  int setsWonByWinner = match.Sets.Count(s =>
      team1Won ? s.Team1Score > s.Team2Score : s.Team2Score > s.Team1Score);
  int totalSets = match.Sets.Count(s => s.Team1Score != s.Team2Score);
  double setRatio = totalSets > 0 ? (double)setsWonByWinner / totalSets : 0.5;
  double gameRatio = (double)winnerUnits / (totalTeam1 + totalTeam2);
  scoreRatio = Math.Clamp(sport!.SetWeight * setRatio + (1 - sport.SetWeight) * gameRatio, 0.5, 1.0);
  ```
- Else: logica esistente invariata
