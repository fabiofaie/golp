## Analisi requisiti

Oggi il caricamento della dashboard richiede fino a 4 round-trip: `GET /circles/me` (lista circoli, innescato da `ActiveCircleService.ensureLoaded()`), `GET /circles/{id}/matches` (ultime partite del circolo attivo, in un `effect()` che dipende dal completamento del primo), `GET /match/mine?status=pending` e `GET /match/mine?status=disputed` (US-068, uniti via `forkJoin`). I primi tre non sono paralleli tra loro: `getMatches(circleId)` non può partire finché `circles/me` non ha risolto l'id del circolo attivo — è una dipendenza dati reale, non solo un limite implementativo.

AC1 chiede "al massimo una chiamata di rete" per popolare l'intera dashboard. Questo richiede un nuovo endpoint di aggregazione lato backend che componga, in un'unica query coordinata: lista circoli (per il selettore), dettaglio del circolo attivo (rating, posizione, ultime partite, conteggio confermate) oppure vista aggregata "tutti i circoli" (US-067: conteggio circoli, partite confirmed totali, % vittorie), e le richieste urgenti cross-circolo (US-068: pending + disputed). Il nuovo endpoint deve quindi sapere se il client è in modalità "circolo singolo" o "tutti i circoli" — riceve un query param opzionale `circleId` (guid del circolo attivo, oppure assente/`all` per la modalità aggregata), analogamente a come il client oggi decide localmente cosa fetchare in base a `ActiveCircleService.activeSelection()`.

AC2 (fallback a chiamate parallele se il single-call non è tecnicamente fattibile) non si applica: il nuovo endpoint rende AC1 raggiungibile direttamente, quindi non serve un percorso alternativo multi-chiamata. AC3 (tempo di caricamento non degrada linearmente col numero di circoli) è soddisfatto per costruzione: il nuovo endpoint calcola i dati pesanti (partite recenti, statistiche) solo per il circolo attivo (o l'aggregato leggero per "tutti i circoli", che è già un conteggio, non un fetch per-circolo), non per ciascun circolo singolarmente — la lista circoli stessa resta O(numero circoli dell'utente), costo già presente oggi e ineliminabile (serve comunque per popolare il selettore). AC4 (fallimento di una porzione non blocca le altre) va gestito **internamente** all'endpoint: le sotto-query rischiose (partite recenti, richieste urgenti) sono avvolte in try/catch che degradano a liste vuote/null invece di far fallire l'intera risposta con 500 — dato che ora è un'unica chiamata, "non bloccare le altre sezioni" si traduce in "un errore parziale non deve abbattere l'intera risposta".

Nessuna ambiguità critica: la forma esatta del contratto risposta è una decisione tecnica (Architect), non un'ambiguità di requisito.

## Soluzione tecnica

Approccio: nuovo endpoint `GET /dashboard/summary?circleId={guid|omesso}` in un nuovo `DashboardEndpoints.cs`, che compone in un'unica risposta JSON tutto ciò che oggi la dashboard richiede con 3-4 chiamate separate. Il frontend sostituisce la sequenza di fetch in `dashboard.component.ts` con una singola chiamata a questo endpoint, mantenendo lo stesso comportamento reattivo (cambio di circolo attivo → nuova chiamata con `circleId` aggiornato).

**Decisioni chiave:**
- Endpoint top-level `/dashboard/summary`, non annidato sotto `/circles/{id}`, perché il payload combina dati cross-circolo (richieste urgenti, lista circoli) con dati per-circolo (rating, ultime partite) — non ha una singola risorsa "circolo" come radice naturale.
- `circleId` come query param opzionale: assente o `all` → payload modalità aggregata (US-067); guid valido e di cui l'utente è membro → payload modalità circolo singolo. Se il guid non è valido o l'utente non è membro, l'endpoint ignora il parametro e ricade sulla modalità aggregata (nessun errore bloccante, coerente con AC4).
- Riuso della logica di calcolo già esistente (`GetMyCirclesAsync`, query pending/disputed di `MyMatchEndpoints`, `GetMatches` di `MatchEndpoints`) tramite chiamata diretta ai metodi privati esistenti dove possibile, o riscrittura minima inline se i metodi non sono facilmente riusabili senza refactor invasivo — **il developer valuta caso per caso in fase di implementazione**, preferendo sempre il riuso a scapito della duplicazione.
- Non si tocca `/circles/me`, `/circles/{id}/matches`, `/match/mine`: restano invariati e continuano a essere usati da altri consumer (`my-circles` page, `active-circle-panel`, `match-history`). Il nuovo endpoint è additivo.
- Resilienza interna: try/catch per-sezione dentro l'handler, non propagato come 500 — ogni sezione del payload può essere `null`/vuota con un flag `xxxAvailable: false` implicito (assenza del campo = non disponibile), la UI la tratta come "nessun dato" invece di bloccare il render.

**Contratto API (nuovo):**

`GET /dashboard/summary?circleId={guid}` (auth required)

```json
{
  "circles": [ /* stessa forma di CircleSummary già esposta da /circles/me */ ],
  "activeCircle": {
    "id": "...", "name": "...", "sport": "...", "myRating": 0, "myRank": 0, "memberCount": 0,
    "recentMatches": [ /* stessa forma di MatchSummary da GET /circles/{id}/matches, filtrate confirmed */ ],
    "confirmedMatchesCount": 0
  },
  "aggregate": null,
  "urgentMatches": [ /* stessa forma di MyMatchSummary, pending+disputed, ordinate per data */ ]
}
```

Quando `circleId` è assente/`all`, `activeCircle` è `null` e `aggregate` è popolato:
```json
"aggregate": { "circlesCount": 0, "confirmedMatchesCount": 0, "winRate": 0 }
```

**Rischi:**
- Nessun blocker aperto (`Blocked by: -`).
- Riscrivere la logica di `GetMyCirclesAsync`/query pending-disputed dentro il nuovo endpoint rischia duplicazione se i metodi esistenti non sono facilmente estraibili in helper condivisi — il developer valuta refactor minimo (estrarre in metodi `internal static` riusabili) solo se il costo è basso, altrimenti duplica con un commento che rimanda alla fonte originale.
- L'endpoint aggregato introduce un secondo punto di manutenzione per la forma dei dati (`CircleSummary`, `MatchSummary`, `MyMatchSummary`) — se il contratto di uno di questi cambia in futuro, va aggiornato anche qui. Accettabile per il guadagno di performance percepita richiesto dalla storia.
