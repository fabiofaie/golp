## Analisi requisiti

Login/register oggi ritornano solo `{ token }`; ogni AC implica cambio di contratto (`{ accessToken, refreshToken }`) su 3 endpoint esistenti (register, login, +nuovo refresh) e impatta `AuthService`/`auth.interceptor.ts` frontend, che oggi gestiscono un solo token in localStorage. Edge case impliciti: revoca su logout deve avvenire lato server (oggi `AuthService.logout()` frontend solo rimuove da localStorage — nessuna chiamata API). Cambio password (`ConfirmPasswordResetAsync`, `AuthEndpoints.cs:114`) deve revocare tutti i refresh token dell'utente — punto di estensione già isolato nel codice esistente. Nessun nuovo permesso: un utente può revocare solo le proprie sessioni.

**Open question del backlog risolta in fase di planning:** rotazione con reuse-detection è incluso nello scope (non rimandato), perché il costo implementativo è modesto riusando il pattern già presente in `PasswordResetService` (hash, mai plaintext persistito) e perché senza rotazione la promessa di sicurezza della storia (revocabilità) è debole.

## Soluzione tecnica

Nuova entità `RefreshToken` (pattern identico a `PasswordResetToken`: hash SHA256 in DB, mai plaintext) con rotazione obbligatoria a ogni uso e reuse-detection. Login/register emettono coppia token; `POST /auth/refresh` consuma il refresh token corrente, lo marca come sostituito, emette coppia nuova (sliding window: scadenza ricalcolata dal momento dell'uso). Riuso di un token già consumato → revoca a cascata di tutta la "famiglia" di rotazioni (stesso `FamilyId`), segnale di furto.

| Componente | Modifica |
|---|---|
| `Data/Entities/RefreshToken.cs` | Nuova entità: `Id, UserId, TokenHash, FamilyId, CreatedAt, LastUsedAt, ExpiresAt, RevokedAt, ReplacedByTokenHash, UserAgent` |
| `Data/AppDbContext.cs` | `DbSet<RefreshToken>`, config `OnModelCreating` (indice univoco su `TokenHash`) |
| Migration | `AddRefreshTokens` |
| `Services/IJwtService` / `JwtService.cs` | Nessuna modifica — access token resta come oggi (durata breve, già configurabile) |
| Nuovo `Services/IRefreshTokenService` / `RefreshTokenService.cs` | `IssueAsync`, `RotateAsync` (con reuse-detection), `RevokeAsync`, `RevokeAllForUserAsync` |
| `Endpoints/AuthEndpoints.cs` | `register`/`login` ritornano `{ accessToken, refreshToken }`; nuovo `POST /auth/refresh`, nuovo `POST /auth/logout`; `ConfirmPasswordResetAsync` chiama `RevokeAllForUserAsync` |
| `appsettings.json` (+ Development/Testing) | `Jwt:RefreshTokenExpiryDays` (default 90) |
| Frontend `auth.service.ts` | Store coppia token, metodo `refresh()`, `logout()` chiama API prima di pulire localStorage |
| Frontend `auth.interceptor.ts` | Intercetta 401, tenta refresh una volta, ripete richiesta originale; se refresh fallisce, redirect login |

**Decisioni chiave:**
- Refresh token plaintext mai persistito (solo hash SHA256), riuso diretto di `PasswordResetService.ComputeSha256`.
- Rotazione obbligatoria a ogni refresh: più sicuro, costo implementativo basso dato il pattern già presente nel progetto.
- `FamilyId` (Guid) condiviso da tutta la catena di rotazioni di una sessione: permette revoca a cascata su reuse-detection senza risalire manualmente la catena.
- Niente tabella "sessioni" separata: `RefreshToken` stesso è il record di sessione (un record attivo = un device loggato).

**Contratto API:**
```
POST /auth/refresh
Body: { refreshToken: string }
200: { accessToken, refreshToken }
401: refresh token scaduto / revocato / non trovato → frontend redirect login

POST /auth/logout
Body: { refreshToken: string }
200: refresh token revocato lato server
```

**Rischi:**
- Refresh concorrenti (es. due tab che refreshano nello stesso istante) possono generare falsi positivi di reuse-detection. Mitigazione possibile: piccola finestra di tolleranza sul token appena ruotato. Per MVP (basso traffico, utenti singoli per device) si accetta il rischio residuo senza finestra di tolleranza, da rivalutare se segnalato.
- Nessun dato da migrare: nuova tabella, nessun campo aggiunto a `User` o `Circle`.
