## Analisi requisiti

### Phase A — Analyst

Due eventi generano una notifica distinta: registrazione nuovo utente (`POST /auth/register`) e creazione nuovo circolo (`POST /circles`). Entrambe le notifiche vanno a `iscrizioni.golp@eqproject.it`, che va esternalizzata in configurazione (non hardcoded) per flessibilità futura.

Il requisito critico è il **fire-and-forget**: un errore SMTP non deve mai bloccare il completamento della richiesta principale. L'endpoint risponde al client prima che l'email sia inviata o fallita; eventuali errori vanno loggati ma ingoiati.

In ambiente di sviluppo il comportamento esistente di `DevelopmentEmailService` (log console) è sufficiente: aggiungere i due nuovi metodi con lo stesso schema. Nessuna ambiguità critica aperta.

**Edge case identificati:**
- SMTP timeout in produzione → log error, operazione principale OK
- Configurazione `StaffNotificationEmail` mancante → log warning, skip notifica silenzioso (non crash startup)
- Registrazione che fallisce validation (email duplicata, password corta) → nessuna notifica (la notifica parte solo dopo `db.SaveChangesAsync()`)

---

## Soluzione tecnica

### Phase B — Architect

**Approccio:** Estendere `IEmailService` con due nuovi metodi di notifica staff. Chiamarli in fire-and-forget (`_ = Task.Run(...)` con catch+log) nei due endpoint già esistenti. Nessuna entità DB nuova, nessuna migrazione, nessun cambio frontend.

**Decisioni chiave:**
- Indirizzo staff in `appsettings.json` → `Notifications:StaffEmail` (override per ambiente)
- Dev: `DevelopmentEmailService` implementa i due metodi con lo stesso `LogAsync` esistente → zero spam reale
- Prod: `SmtpEmailService` usa template HTML minimal dedicati

**Componenti toccati:**

| File | Modifica |
|------|----------|
| `Services/IEmailService.cs` | +2 metodi |
| `Services/SmtpEmailService.cs` | +2 implementazioni |
| `Services/DevelopmentEmailService.cs` | +2 implementazioni |
| `EmailTemplates/new-user-notification.html` | nuovo template |
| `EmailTemplates/new-circle-notification.html` | nuovo template |
| `Endpoints/AuthEndpoints.cs` | inietta `IEmailService`, fire-and-forget dopo SaveChanges |
| `Endpoints/CircleEndpoints.cs` | inietta `IEmailService` + `ILogger`, fire-and-forget dopo SaveChanges |
| `appsettings.json` | +`Notifications:StaffEmail` |

**Fire-and-forget pattern:**
```csharp
_ = emailService.SendNewUserNotificationAsync(user.Email, user.Name, DateTime.UtcNow)
    .ContinueWith(t => logger.LogError(t.Exception, "Staff notification failed"),
                  TaskContinuationOptions.OnlyOnFaulted);
```

**Rischi:**
- `SmtpEmailService` ctor lancia `InvalidOperationException` se `Smtp:Host` mancante — ok in prod, ok in dev perché viene registrato `DevelopmentEmailService`
- Nessun rischio di race condition: fire-and-forget non modifica stato condiviso
