## Analisi requisiti

La storia richiede tre comportamenti osservabili: (1) un punto d'ingresso dove il super admin inserisce un'email, (2) l'app che opera "come" l'utente target dopo la sottomissione, (3) un indicatore permanente e visibile durante l'impersonazione. Riuso deliberato dell'infrastruttura JWT esistente: l'impersonazione emette un token normale per l'utente target (stesso meccanismo di login), quindi tutto lo scoping `circle_id` esistente continua a funzionare senza modifiche — non serve toccare `MatchEndpoints`/`CircleEndpoints`. Il rischio di privilege escalation è concentrato in un solo punto: l'endpoint che emette il token deve richiedere il claim `super_admin` (già disponibile da US-058) e non deve mai accettare una password del target.

Edge case: email inesistente → 404 chiaro, nessun token; email che corrisponde ma appartiene proprio al super admin (auto-impersonazione) → permesso ma innocuo (il banner apparirebbe comunque, comportamento accettato, non è un caso da bloccare esplicitamente); un secondo super admin che tenta di impersonare mentre è già in corso una sessione impersonata di un altro admin — fuori scope, ogni sessione browser è indipendente. Requisito implicito: il nuovo token deve portare traccia di CHI ha avviato l'impersonazione (claim `impersonator_id`), perché senza questo dato l'audit log di US-060 non potrebbe funzionare — lo aggiungo qui anche se l'audit vero e proprio è task di US-060, per non dover ritoccare il contratto del token in due storie consecutive.

Nessuna ambiguità critica bloccante. Un punto da segnalare: la storia non specifica se durante l'impersonazione deve restare disponibile un refresh token — assumo di sì (stesso comportamento del login normale), altrimenti la sessione impersonata scadrebbe troppo in fretta per un uso di supporto reale.

## Soluzione tecnica

Nuovo endpoint `POST /admin/impersonate` in `AdminEndpoints.cs`, protetto dal claim `super_admin` (stesso pattern di `/admin/whoami`). Riceve `{ email }`, cerca l'utente per email, se trovato genera un token via `IJwtService` con un nuovo parametro opzionale `impersonatorId` che aggiunge il claim `impersonator_id` al JWT — così il token del target porta sia l'identità del target (sub/email standard) sia la provenienza (chi impersona). Riuso `IRefreshTokenService.IssueAsync` per il refresh token, tenendolo legato all'utente target come in un login normale.

Frontend: nuova pagina `admin/impersonate` (solo super admin, guard dedicata) con form email. `AuthService` guadagna `isImpersonating()` (decodifica claim `impersonator_id` dal token corrente) e `startImpersonation(email)` (chiama l'endpoint, sostituisce token/refreshToken in localStorage, naviga a `/dashboard`). Un componente `ImpersonationBannerComponent`, montato nell'app shell (root component, sempre visibile se autenticato), mostra "Stai impersonando [email/nome]" quando `isImpersonating()` è true — nessuna logica di uscita qui (arriva in US-060).

| Componente | Modifica |
|---|---|
| `Services/IJwtService.cs`, `JwtService.cs` | `GenerateToken` accetta `impersonatorId` opzionale, claim `impersonator_id` condizionale |
| `Endpoints/AdminEndpoints.cs` | + `POST /admin/impersonate` |
| `auth/auth.service.ts` | + `isImpersonating()`, `startImpersonation(email)` |
| `app/shared/impersonation-banner/` (nuovo) | componente banner |
| `app.component.ts`/`.html` | monta il banner nello shell |
| `app.routes.ts` | + route `admin/impersonate` con guard super-admin |
| `auth/super-admin.guard.ts` (nuovo) | guard che legge claim `super_admin` dal token, redirect se assente |

Contratto API: `POST /admin/impersonate` body `{ email }` → 200 `{ accessToken, refreshToken, token }` (stesso shape di login/register, per riuso diretto in `AuthService`), 404 se email non trovata, 403 se il chiamante non è super admin, 401 senza token.

Rischi: il claim `impersonator_id` nel JWT è testo semplice leggibile lato client (base64, non cifrato) — accettabile perché non è un segreto, serve solo per UI/audit, mai per decisioni di autorizzazione lato client. Il guard frontend `super-admin.guard.ts` è solo UX (nasconde la pagina); la vera autorizzazione resta lato server sull'endpoint — da non confondere le due cose in review.
