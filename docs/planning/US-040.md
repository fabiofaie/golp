## Analisi requisiti

US-040 introduce un flusso di conferma partita senza login, tramite token temporaneo UUID inviato via email. Il token identifica univocamente la coppia (Match, User) e ha validità 72h. I requisiti sono chiari: nuova tabella `MatchConfirmationTokens`, 3 endpoint pubblici (`GET /m/{token}`, `POST /m/{token}/confirm`, `POST /m/{token}/dispute`), pagina frontend `/m/{token}` senza `authGuard`.

Requisiti impliciti: le email già inviate in `CreateMatchAsync` devono puntare al link token-specifico `/m/{token}` invece del generico `/circles/{circleId}/matches/{matchId}`. Per gli ospiti phone-only (no email) il token viene comunque generato in DB (utile per US-042 WhatsApp), ma nessuna email inviata. La logica di conferma/disputa deve delegare all'esatta stessa logica dell'endpoint autenticato esistente, per garantire equivalenza effetti su `Match.Status` e rating ELO.

Edge case rilevanti: token scaduto → 410 Gone; token già usato → risposta con stato attuale (non errore bloccante, l'utente ha già risposto); match già `confirmed/disputed` quando il token viene usato → 409 Conflict; race condition sulla 4a conferma → già gestita dalla logica esistente con count check atomico.

---

## Soluzione tecnica

**Approccio:** aggiunta minimale al sistema esistente. Nuova entity + nuovo endpoint group pubblico + generazione token in `CreateMatchAsync` + aggiornamento URL email. Zero duplicazione della logica di conferma: i nuovi endpoint la delegano agli stessi helper privati del `MatchEndpoints`.

**Decisioni chiave:**

| Decisione | Scelta | Alternativa scartata |
|---|---|---|
| Token storage | Guid raw (non hashato) | Hash SHA256 come PasswordResetToken |
| Motivo | Token nell'URL è già il segreto; i log server non lo espongono in produzione. Hashing aggiunge complessità senza beneficio reale per questo use case | Hash sarebbe più sicuro ma over-engineering per un token pubblico a 72h |
| Generazione token | Sincrona + stessa `SaveChangesAsync` del match | Fire-and-forget async |
| Motivo | Atomicità: se la save fallisce, nessun token orfano. Il fire-and-forget legge token già persistiti | Asincrono crea finestra di race condition |
| Endpoint prefix | `/m/{token}` (root, non `/api/m`) | `/api/m/{token}` |
| Motivo | AC esplicito: `golp.app/m/{token}`, URL corta per email e WhatsApp | |
| DRY logica confirm | Estrarre helper `static PerformConfirmAsync` e `PerformDisputeAsync` in `MatchEndpoints` | Duplicare il codice |

**Contratto API:**

```
GET /m/{token}
→ 200 { match: { sport, circleName, team1[], team2[], sets[], winnerTeam, status, confirmationsCount },
         token: { valid: true, userId, userHasConfirmed, userName } }
→ 404 { error: "Token non trovato" }
→ 410 { error: "Token scaduto" }
→ 400 { error: "Token già utilizzato" }  // già usato ma non scaduto → info + stato attuale

POST /m/{token}/confirm  (body: vuoto)
→ 200 { status, confirmationsCount, isActivated: bool }
→ 404 | 410 | 409 { error: "..." }

POST /m/{token}/dispute  (body: vuoto)
→ 200 { status, isActivated: bool }
→ 404 | 410 | 409 { error: "..." }
```

### TASK-3 — PublicMatchEndpoints + wire in Program.cs ✅ (2026-06-30 14:30)

**File:**
- `src/Golp.Api/Endpoints/PublicMatchEndpoints.cs` (NUOVO)
- `src/Golp.Api/Program.cs`
- `src/Golp.Api/Endpoints/MatchEndpoints.cs` (refactor helper)

**Acceptance:**

`GET /m/{token}` — nessun `RequireAuthorization()`:
- Lookup `MatchConfirmationToken` by `Token` value
- Se non trovato → 404
- Se `ExpiresAt < now` → 410 `{ error: "Token scaduto" }`
- Se `UsedAt != null` → 200 con flag `{ tokenUsed: true, ... }` + dati match (user può vedere esito)
- Altrimenti → 200 con `{ match: {...}, token: { valid: true, userId, userHasConfirmed } }`
- `match` include: sport (da `circle.Sport`), `circleName`, team1[], team2[] (con `name`, `isActivated`), `sets[]`, `winnerTeam`, `status`, `confirmationsCount`
- `userHasConfirmed` = esiste `MatchConfirmation` per `(matchId, userId)`

`POST /m/{token}/confirm`:
- Stessa validazione token GET
- Se `UsedAt != null` → 200 `{ alreadyDone: true, status, confirmationsCount }` (idempotente)
- Delegate a helper `PerformConfirmAsync(matchId, userId, db, ratingService, pushService)`
- Segna `token.UsedAt = DateTimeOffset.UtcNow`
- `SaveChangesAsync`
- Return `{ status, confirmationsCount, isActivated: user.IsActivated }`

`POST /m/{token}/dispute`:
- Stesso pattern confirm
- Delegate a helper `PerformDisputeAsync(matchId, userId, db, ...)`
- Return `{ status, isActivated }`

**Helper refactor:** estrarre logica da `ConfirmMatchAsync` e `DisputeMatchAsync` in `private static async Task<...> PerformConfirmAsync(...)` e `PerformDisputeAsync(...)`. Gli endpoint autenticati esistenti delegano agli stessi helper.

**Program.cs:** `app.MapPublicMatchEndpoints()` (senza `RequireAuthorization`)

**Verifica:** `dotnet build` green; `GET /m/invalid-guid` → 404.
