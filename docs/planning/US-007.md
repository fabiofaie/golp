## Analisi requisiti

**AC e comportamenti concreti:**
1. **ELO alla transizione confirmed** — trigger: `Match.Status` passa a `confirmed` → `IRatingService.CalculateAndApplyAsync` invocato sincrono nella stessa transazione. Se il calcolo fallisce, la conferma non è persistita.
2. **K=48 cold start (<15 partite confermate)** — per ognuno dei 4 giocatori, il servizio conta le partite `confirmed` nel circolo PRIMA di questo match. Se < 15, K=48; altrimenti K=32.
3. **score_ratio = somma unità** — `totalA = sum(MatchSet.Team1Score)`, `totalB = sum(MatchSet.Team2Score)`, `score_ratio_A = totalA / (totalA + totalB)`. Questa sostituisce il binario 0/1 standard ELO.
4. **Delta per giocatore salvato sulla partita** — 4 campi nullable `int?` su `Match` (uno per player). Usati da US-009 per mostrare `+N pt`. Null = partita non ancora confermata o disputed.
5. **Pending/disputed → nessun ricalcolo** — guard `if (match.Status != "confirmed") return` all'ingresso del servizio.
6. **Idempotenza** — guard `if (match.DeltaTeam1Player1 != null) return` prima del calcolo. Delta già presenti = già processato.
7. **Rating iniziale 1000** — garantito da `CircleMembership.Rating = 1000` (già implementato).

**Edge case identificati:**
- `totalA + totalB == 0` → impossibile per validazione US-004 (punteggio obbligatorio con vincitore), ma guard divisione per zero necessario per robustezza.
- K è per singolo giocatore: due giocatori dello stesso team possono avere K diversi (uno in cold start, l'altro no).
- Il conteggio partite per cold start esclude il match corrente (anche se già confirmed): snapshot del count prima del commit.

**Requisiti impliciti:**
- Il servizio NON chiama `SaveChangesAsync` internamente — il caller (US-005) gestisce la transazione unificata: [conferm match → aggiorna rating → SaveChanges].
- `IRatingService` registrato come Scoped in DI per poter iniettare `AppDbContext`.

---

## Soluzione tecnica

**Approccio:** Servizio `RatingService` con interfaccia `IRatingService`, iniettato nel punto di conferma (US-005). Calcolo sincrono nella stessa transazione di conferma per consistenza. Delta salvati su `Match` come 4 campi int? nullable.

**Componenti toccati:**

| Layer | Componente | Modifica |
|-------|-----------|----------|
| Backend | `Match.cs` | aggiunge `DeltaTeam1Player1, DeltaTeam1Player2, DeltaTeam2Player1, DeltaTeam2Player2` (int?) |
| Backend | `IRatingService.cs` | **new** interfaccia |
| Backend | `RatingService.cs` | **new** implementazione formula ELO |
| Backend | `Program.cs` | `services.AddScoped<IRatingService, RatingService>()` |
| Backend | EF Migration | `AddMatchEloDeltas` |
| Backend | US-005 (futuro) | chiama `IRatingService.CalculateAndApplyAsync` dopo transizione a confirmed |

**Formula ELO (da PRD §Algoritmo):**

```
teamRating_A  = (rating_A1 + rating_A2) / 2.0
teamRating_B  = (rating_B1 + rating_B2) / 2.0
expected_A    = 1.0 / (1.0 + Math.Pow(10.0, (teamRating_B - teamRating_A) / 400.0))
score_ratio_A = (double)totalUnitsA / (totalUnitsA + totalUnitsB)
K_per_player  = (confirmedMatchCount_player < 15) ? 48 : 32
delta_A       = (int)Math.Round(K_per_player * 0.7 * (score_ratio_A - expected_A))
delta_B       = -(int)Math.Round(K_per_player_B * 0.7 * (score_ratio_A - expected_A))
```

> ⚠️ `amplifier = 0.7` e `K = 32/48` sono hardcoded (out of scope tuning via config, da PRD).

**Decisioni chiave:**
- **Sincrono in transazione** — se il calcolo ELO lancia eccezione, `SaveChangesAsync` non viene chiamato e la conferma non viene persistita. Consistenza garantita. Tradeoff: latenza leggermente più alta alla 4ª conferma.
- **4 campi nullable su Match vs tabella MatchPlayerDelta** — 4 campi sono sufficienti per il modello fisso 2v2; una tabella separata sarebbe over-engineering per MVP.
- **arrotondamento `Math.Round` a int** — `CircleMembership.Rating` è int; delta arrotondato per semplicità.
- **K indipendente per giocatore** — due player del medesimo team possono avere K diversi; il delta risultante per i due membri del team può quindi differire leggermente.

**Rischi:**
- Integrazione con US-005 è dipendenza esterna: `RatingService` è indipendente ma non verrà mai chiamato finché US-005 non è implementata. Il piano di US-005 deve importare `IRatingService`.
- Formula: verificare `docs/PRD.md §Algoritmo` al momento dell'implementazione (il testo del PRD non è stato letto direttamente in questa pianificazione).
