## Analisi requisiti

Le 7 AC si dividono in due meccanismi distinti che vanno tenuti separati a livello di componenti: (1) **detection + gating** — rilevare se l'utente è già in modalità standalone (PWA installata) o ha già visto/chiuso l'invito, per decidere se mostrare qualcosa; (2) **contenuto** — quale messaggio/guida mostrare in base a OS+browser rilevati. Il rilevamento standalone su iOS Safari richiede `navigator.standalone` (proprietà non standard, non coperta da `display-mode: standalone`), mentre su Android Chrome basta `matchMedia('(display-mode: standalone)')` — sono due API diverse per lo stesso concetto, vanno entrambe gestite nello stesso check.

Edge case rilevanti: (a) l'evento `beforeinstallprompt` va catturato e bufferizzato **appena disponibile** (può arrivare prima che l'utente interagisca con la guida, e il browser lo invia una sola volta — se non lo si salva subito si perde l'occasione di triggerare il prompt nativo su azione utente successiva); (b) su iOS Safari `beforeinstallprompt` non esiste affatto — l'unica via è manuale (Condividi → Aggiungi a Home), va sempre mostrata la guida testuale lì; (c) il "non più mostrato nelle sessioni successive" (AC2) richiede persistenza in `localStorage` (non `sessionStorage`, altrimenti riapparirebbe ad ogni nuova tab/sessione) con due distinti motivi di sopravvivenza: utente ha chiuso il banner, oppure l'app è già installata (rilevato a runtime, non serve flag per questo caso).

Requisito implicito non esplicitato in AC ma necessario per AC7 (gestione desktop esplicita): la story lascia aperta la domanda se mostrare la guida anche su desktop. Decisione presa in fase Architect (sotto) per non bloccare il piano: **banner/guida solo su dispositivi mobile** (rilevati via UA + assenza di puntatore fine `(pointer: fine)` come fallback), desktop sempre silenzioso. Motivo: la storia descrive esplicitamente "Marco... da un browser mobile" come persona, e i due Out-of-scope non menzionano desktop come requisito — trattarlo come fuori scope minimizza superficie di test mantenendo AC7 soddisfatto (comportamento esplicito = "non mostrare nulla").

## Soluzione tecnica

Nuovo modulo `frontend/golp-app/src/app/shared/pwa-install/`, stesso pattern già usato per `shared/update/` (service + componente standalone, wiring in `app.component.ts`). Nessuna modifica backend: tutta la logica è client-side (UA sniffing, `localStorage`, API browser standard/non-standard). Tre pezzi: un service di detection puro (OS/browser/standalone), un service di stato/evento (dismissed flag + cattura `beforeinstallprompt`), due componenti UI (banner compatto + guida a step).

**Decisioni chiave:**
- **UA sniffing via `navigator.userAgent`, non `navigator.userAgentData`** — `userAgentData` non è supportato su Safari/iOS (proprio il caso più critico da rilevare), quindi va escluso come fonte primaria. Tradeoff: UA string è fragile a lungo termine (può cambiare), ma è l'unica via cross-browser oggi disponibile per questo caso d'uso.
- **Banner solo mobile, desktop silenzioso** (vedi Analisi requisiti) — riduce scope, ma rimanda il caso desktop a una storia futura se richiesto.
- **`beforeinstallprompt` catturato a livello di service singleton, bufferizzato in un `BehaviorSubject`** — disponibile per la guida anche se l'evento arriva prima che l'utente apra la guida.
- **Config statica per coppia OS+browser** (non logica condizionale sparsa nei componenti) — un piccolo dizionario `{ios: {safari: [...steps]}, android: {chrome: [...], samsung: [...], firefox: [...]}}` con fallback generico se combinazione non riconosciuta (soddisfa AC6).

**Componenti toccati:**

| Componente | Tipo modifica |
|---|---|
| `shared/pwa-install/pwa-platform.service.ts` | nuovo — detection OS/browser/standalone-mode |
| `shared/pwa-install/pwa-install.service.ts` | nuovo — stato dismissed (localStorage) + cattura `beforeinstallprompt` |
| `shared/pwa-install/pwa-install-steps.ts` | nuovo — config statica step per OS+browser |
| `shared/pwa-install/pwa-install-banner.component.ts` | nuovo — banner invito + apertura guida |
| `shared/pwa-install/pwa-install-guide.component.ts` | nuovo — mini guida a step, azione "Installa" se prompt nativo disponibile |
| `app.component.ts` / `.html` | modifica — wiring banner (stesso pattern di `AppUpdateBannerComponent`) |

Nessuna modifica al modello dati né a contratti API (storia 100% frontend).

**Rischi:**
- UA sniffing per Samsung Internet/Firefox Android può essere impreciso (entrambi basati su Chromium/Gecko con UA string variabile) — mitigato da fallback generico Android se non riconosciuto con certezza.
- `beforeinstallprompt` non garantito ovunque (Chrome può non spararlo se criteri PWA install non soddisfatti, es. manifest incompleto) — la guida testuale resta valida comunque, l'azione diretta è solo un'aggiunta opportunistica.

## Task

- **TASK-1**: `PwaPlatformService` — detection `os: 'ios'|'android'|'other'`, `browser: 'safari'|'chrome'|'samsung'|'firefox'|'other'`, `isStandalone(): boolean` (gestisce sia `navigator.standalone` iOS che `matchMedia`). File: `shared/pwa-install/pwa-platform.service.ts` + `.spec.ts`. Acceptance: detection corretta su UA string di test per le combinazioni note (Safari iOS, Chrome Android, Samsung Internet, Firefox Android, Chrome desktop). Verifica: unit test con UA mockate.
- **TASK-2**: `pwa-install-steps.ts` — config statica `Record<Os, Record<Browser, InstallStep[]>>` con fallback `default` per combinazioni non mappate. File: `shared/pwa-install/pwa-install-steps.ts`. Acceptance: ogni combinazione (ios/safari, android/chrome, android/samsung, android/firefox, fallback) ha almeno 1 step non vuoto. Verifica: unit test su struttura dati.
- **TASK-3**: `PwaInstallService` — cattura `window.addEventListener('beforeinstallprompt', ...)` con `event.preventDefault()` e buffer in `BehaviorSubject<Event|null>`; `shouldShowBanner()` combina `!isStandalone() && !dismissedFlag && isMobile`; `dismiss()` scrive in `localStorage`; `triggerNativePrompt()` chiama `.prompt()` sull'evento bufferizzato se presente. File: `shared/pwa-install/pwa-install.service.ts` + `.spec.ts`. Acceptance: AC1, AC2, AC5, AC7 coperti a livello di stato. Verifica: unit test con `localStorage` mockato e evento simulato via `dispatchEvent`.
- **TASK-4**: `PwaInstallBannerComponent` — banner compatto, due azioni ("Scopri come" → apre guida, "Chiudi" → `dismiss()`), visibile solo se `shouldShowBanner()`. File: `shared/pwa-install/pwa-install-banner.component.ts/.html/.scss` + `.spec.ts`. Acceptance: AC1, AC2. Verifica: unit test su render condizionale e chiamata dismiss.
- **TASK-5**: `PwaInstallGuideComponent` — riceve step da `pwa-install-steps.ts` in base a `PwaPlatformService`, mostra lista step; se `triggerNativePrompt` disponibile mostra bottone azione diretta oltre alla spiegazione manuale. File: `shared/pwa-install/pwa-install-guide.component.ts/.html/.scss` + `.spec.ts`. Acceptance: AC3, AC4, AC5, AC6. Verifica: unit test con platform mockata per ogni combinazione + caso fallback.
- **TASK-6**: Wiring in `app.component.ts`/`.html` — import `PwaInstallBannerComponent`, stesso punto di mount del banner update (no conflitto visivo: banner update e banner install non vanno mostrati sovrapposti, va deciso ordine/priorità se entrambi attivi). Acceptance: banner appare su mobile al primo load se condizioni soddisfatte. Verifica: test wiring su `AppComponent` (pattern già usato per `AppUpdateService`).
- **TASK-7**: Unit test mirati AC7 (desktop) — `shouldShowBanner()` ritorna `false` su UA desktop anche se non standalone e non dismissed. Verifica: estende `.spec.ts` di TASK-3.
- **TASK-8**: E2E Playwright — scenario "primo accesso mobile Android Chrome → banner visibile → click guida → step mostrati → dismiss → reload → banner non più visibile". File: `e2e/pwa-install.spec.ts`. Acceptance: copre il flusso end-to-end principale (AC1, AC2, AC4).

## Test strategy

| AC | Test |
|---|---|
| AC1 (banner al primo accesso mobile non-standalone) | Unit `pwa-install.service.spec.ts` (TASK-3) + E2E (TASK-8) |
| AC2 (non ripetuto dopo chiusura/installazione) | Unit `pwa-install.service.spec.ts` (TASK-3) + E2E reload (TASK-8) |
| AC3 (detection browser/OS) | Unit `pwa-platform.service.spec.ts` (TASK-1) |
| AC4 (guida per combinazione OS+browser) | Unit `pwa-install-guide.component.spec.ts` (TASK-5) + E2E (TASK-8) |
| AC5 (azione diretta se `beforeinstallprompt` disponibile) | Unit `pwa-install.service.spec.ts` + `pwa-install-guide.component.spec.ts` (TASK-3, TASK-5) |
| AC6 (fallback su combinazione non riconosciuta) | Unit `pwa-install-steps.ts` data test (TASK-2) + guide component con platform 'other' (TASK-5) |
| AC7 (desktop gestito esplicitamente) | Unit dedicato (TASK-7) |

**Strategia e2e**: un solo scenario Playwright (TASK-8) che copre il caso più rappresentativo (Android Chrome, perché supporta sia banner che prompt nativo). I casi iOS Safari (no `beforeinstallprompt`) e fallback restano a livello unit — emularli in e2e richiederebbe override UA complessi con basso ritorno rispetto al costo.

**Demo video**: non necessario. La storia è UI ma di basso rischio visivo (banner + lista testuale), già coperta da e2e + screenshot statici nei mockup; un video aggiunge poco sopra agli step della guida già testabili in unit/e2e.
