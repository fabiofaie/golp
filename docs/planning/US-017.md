## Analisi requisiti

Il form simulatore permette di inserire il punteggio reale (non solo vittoria/sconfitta), consentendo il calcolo esatto dello `score_ratio` come fa il backend. L'utente sceglie la modalità risultato:
- **Risultato unico:** due campi (es. 10 e 6) → `score_ratio = max_punti / (punti1 + punti2)`, clampato a [0.5, 1.0]
- **Per set:** coppie dinamiche (es. 6-4, 7-5) → `score_ratio` calcolato sulla somma totale dei punti su tutti i set, identico alla logica backend senza `SetWeight` (poiché il simulatore non conosce lo sport)

Il K dinamico (48 per le prime 15 partite, 32 dopo) dipende dallo storico partite del giocatore nel circolo — informazione non disponibile nel simulatore. Si usa K=32 fisso (caso stabile), con nota nell'UI.

AC1: la route `/elo-info` **non deve avere `canActivate: [authGuard]`** — deve essere accessibile anche a chi non è loggato (es. qualcuno che vuole capire il sistema prima di registrarsi).

## Soluzione tecnica

**Principio:** unica source of truth per la formula ELO è `RatingService.cs`. Il simulatore chiama un nuovo endpoint `POST /simulate-match` nel backend. Nessuna replica della formula in TypeScript: se l'algoritmo cambia nel backend, il simulatore si aggiorna automaticamente senza modifiche frontend.

**Componenti toccati:**

| File | Modifica |
|---|---|
| `src/Golp.Api/Endpoints/SimulateEndpoints.cs` | nuovo — `POST /simulate-match` pubblico (no auth) |
| `src/Golp.Api/Services/RatingService.cs` | estrai logica pura in metodo statico/interno riusabile da SimulateEndpoints |
| `src/Golp.Api/Program.cs` | registra `app.MapSimulateEndpoints()` |
| `app/elo-info/elo-info.component.ts/.html/.scss` | nuovo componente standalone Angular |
| `app/circles/elo-info/elo-info.service.ts` | HttpClient wrapper per `/simulate-match` |
| `app/app.routes.ts` | aggiungi route `/elo-info` senza auth guard |
| `circles/circle-leaderboard/circle-leaderboard.component.html` | aggiungi link in fondo |
| `dashboard/dashboard.component.ts` | aggiungi link "Simula una partita" |

**Contratto API `POST /simulate-match` (pubblico, no JWT):**
```json
// Request
{
  "team1": { "player1Rating": 1050, "player2Rating": 980 },
  "team2": { "player1Rating": 1100, "player2Rating": 1090 },
  "sets": [
    { "team1Score": 10, "team2Score": 6 }
  ],
  "experienced": true   // true = K 32 (>15 partite), false = K 48 (<15 partite)
}

// Response 200
{
  "deltaTeam1Player1": 12,
  "deltaTeam1Player2": 12,
  "deltaTeam2Player1": -12,
  "deltaTeam2Player2": -12
}
```

**Note implementazione backend:**
- `RatingService.CalculateAndApplyAsync` è legato a EF/DB. Estrarre la logica pura di calcolo ELO in un metodo statico `RatingService.ComputeDeltas(...)` — chiamato sia da `CalculateAndApplyAsync` che da `SimulateEndpoints`.
- K=32 fisso nel simulatore (indicato nella response o nella nota UI). Non conosciamo lo storico partite del giocatore simulato.
- Endpoint pubblico: nessun `[Authorize]` / `RequireAuthorization()`.
- Aggiungere il path `/simulate-match` al `proxy.conf.json` del frontend Angular.

**Rischi:** estrarre logica da `RatingService` richiede refactor controllato — coprire con test esistenti prima di toccare.
