## Analisi requisiti

Bug riscontrato su caso reale: partita 6-4, 6-4, 1-6 — la squadra vince 2 set su 3 (match vinto secondo la logica esistente di conferma match, che determina `Match.WinnerTeam` in base ai set, invariata e fuori scope qui) ma la somma game totale (13 vs 16) è sfavorevole. La formula attuale (`basePoints = gameDiffTotale + 1`) usa la somma game totale come proxy di dominio, che può contraddire il vincitore reale per sport a set: un set perso largamente (1-6) può ribaltare il segno della differenza game anche quando la squadra ha vinto la maggioranza dei set. Ogni AC è mappato: AC1/AC2 definiscono la nuova formula e la sua equivalenza per sport senza set; AC3 conferma nessun impatto sui dati storici (append-only, già garantito dal guard di idempotenza esistente in `CalculateAndApplyAsync`); AC4/AC5/AC6 estendono la stessa formula al simulatore, correggendo anche il bug collegato del calcolo lato client di `team1Won` (basato su somma game invece che su set vinti); AC7 aggiorna il testo esplicativo della pagina.

Edge case: sport senza set (basket2v2, burraco) e modalità "risultato unico" hanno un solo elemento nella lista set — la formula deve collassare esattamente sul valore attuale (game diff + 1) per non introdurre regressioni sui match già confermati con quei metodi. Ambiguità risolta: la determinazione di chi ha vinto il match nel simulatore va spostata sul backend (calcolata dai set stessi, maggioranza set vinti), eliminando la duplicazione di logica lato client che ha causato il bug originale — single source of truth. Per il calcolo reale (`CalculateAndApplyAsync`), `Match.WinnerTeam` resta la fonte di verità esistente (stabilita al momento della conferma match, non toccata da questa storia): la nuova formula la usa come dato di input, non la ricalcola.

## Soluzione tecnica

Approccio: sostituire il calcolo di `basePoints` in `GameBonusRatingService` con una versione che opera sulla lista dei set (non più sulla somma totale), mantenendo firma pubblica compatibile dove possibile ma cambiando la sorgente dati di `ComputeMatchPoints`. Il simulatore riusa la stessa funzione pura, eliminando il campo `team1Won` dalla request (calcolato lato server dai set) per chiudere la classe di bug "client calcola male chi ha vinto".

Decisioni chiave:
- **`ComputeMatchPoints` cambia firma**: da `(int winnerUnits, int loserUnits, double winnerScore, double loserScore)` a `(IReadOnlyList<(int Team1, int Team2)> sets, bool team1Won, double winnerScore, double loserScore)`. Motivazione: la nuova formula ha bisogno dei singoli set (per calcolare `setsWon`/`setsLost` e il game-diff dei soli set vinti), non più di due totali aggregati. Trade-off: firma pubblica cambia, tutti i chiamanti (servizio reale + simulatore + test) vanno aggiornati nella stessa storia — accettabile, nessun consumer esterno.
- **Simulatore: winner derivato dai set, non passato dal client**: `SimulateGameBonusRequest` perde `Team1Won`; il backend calcola chi ha vinto contando i set vinti da ciascuna squadra (maggioranza). Elimina la duplicazione di logica che ha causato il bug (`totalMine > totalOpp` nel componente Angular). Per modalità "risultato unico" il calcolo collassa naturalmente (1 solo "set", vince chi ha lo score più alto in quell'unico elemento).
- **`CalculateAndApplyAsync` non cambia la fonte del vincitore**: continua a usare `match.WinnerTeam` (invariato, stabilito alla conferma match da logica esistente non in scope). Cambia solo come i punti vengono calcolati dato quel vincitore.
- **Nessuna migrazione dati**: `Match.GameBonusWinnerPoints` resta append-only, i match già confermati non vengono ricalcolati (stesso pattern di idempotenza di US-052, invariato).
