**Query backend (schema LINQ):**
```csharp
var confirmedCounts = await db.Matches
    .Where(m => m.CircleId == id && m.Status == "confirmed")
    .SelectMany(m => new[] { m.Team1Player1Id, m.Team1Player2Id, m.Team2Player1Id, m.Team2Player2Id })
    .GroupBy(uid => uid)
    .Select(g => new { UserId = g.Key, Count = g.Count() })
    .ToDictionaryAsync(x => x.UserId, x => x.Count);

var members = await db.CircleMemberships
    .Where(m => m.CircleId == id)
    .Select(m => new { m.UserId, Name = m.User.Name, m.Rating })
    .ToListAsync();

var classified = members
    .Where(m => confirmedCounts.ContainsKey(m.UserId))
    .OrderByDescending(m => m.Rating)
    .ThenByDescending(m => confirmedCounts[m.UserId])
    .Select((m, i) => new { userId = m.UserId, name = m.Name, rating = m.Rating, rank = i + 1, confirmedMatches = confirmedCounts[m.UserId] })
    .ToList();

var unclassified = members
    .Where(m => !confirmedCounts.ContainsKey(m.UserId))
    .Select(m => new { userId = m.UserId, name = m.Name })
    .ToList();
```

**Decisioni chiave:**
- **Endpoint separato da GET /circles/{id}/members** â€” semantica diversa; members serve il browse, leaderboard serve la classifica con confirmed split.
- **isCurrentUser in frontend** â€” evita di portare il userId nel response (giĂ  disponibile in AuthService).
- **confirmedMatches calcolato in DB** â€” non in memoria, query su Matches filtrata per circolo e status.

**Rischi:**
- US-007 non DONE â†’ tutti a rating 1000, classifica poco significativa (ma funzionale).
- La query `SelectMany` su 4 colonne Ă¨ un pattern insolito in EF â€” testare performance con dataset medio.
