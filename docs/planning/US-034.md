## Analisi requisiti

US-034 corregge il calcolo ELO per due casi oggi mal gestiti negli sport a set (padel, beachtennis): pareggio di set con game a decidere, e pareggio di game con set a decidere. La barriera principale era la validazione in `MatchEndpoints.cs:83-84` che rigetta con 400 qualunque pareggio di set, raggiungendo il vincitore solo tramite `team1Wins > team2Wins`. Va sostituita da una validazione a due livelli: se i set sono pari, decide il totale game; se anche i game sono pari, resta vero pareggio → 400 (comportamento invariato, fuori scope).

Edge case rilevanti: (1) `RatingService.CalculateAndApplyAsync` oggi usa `match.WinnerTeam` per decidere `winnerUnits` — la logica branch "set pari → solo game" e "game pari → solo point_unit" deve riusare lo stesso branch già esistente per `useBlended` (linee 101-119), aggiungendo i due sotto-casi. (2) Round-to-zero: `ComputeDeltas` arrotonda con `Math.Round` — serve un floor di ±1 quando `margin != 0` ma il delta arrotondato è 0, applicato coerentemente per ogni giocatore (segni opposti). (3) Sport senza set restano invariati per esplicita decisione (Out of scope), quindi nessuna modifica al ramo `circle.Sets == false`.

Nessuna ambiguità critica residua — risolte con l'utente prima del plan (pareggio di set ammesso solo se i game decidono; pareggio totale resta bloccato per tutti gli sport).

## Soluzione tecnica

Approccio: estendere la validazione di creazione partita per ammettere il pareggio di set quando i game decidono (nuovo ramo, vincitore = squadra con più game), e arricchire `RatingService` con un calcolo di `scoreRatio` dedicato ai due casi di pareggio parziale, riusando `useBlended` come base. Il floor ±1 si applica a `ComputeDeltas`, non al ramo `scoreRatio`, perché è indipendente da quale formula (blended o non) ha prodotto il margine.

**Decisioni chiave:**
- Calcolo `scoreRatio` per "set pari, game decidono": ignora `setRatio`, usa solo `gameRatio` (equivalente a peso set = 0 per quella partita specifica) — non serve nuovo branch nella formula, basta forzare il peso set effettivo a 0 quando `setsWonByWinner == totalSets - setsWonByWinner`.
- Calcolo per "game pari, set decidono": simmetrico, forza il peso set effettivo a 1 (assoluto) quando `totalTeam1 == totalTeam2`.
- Floor ±1: applicato in `DeltaFor` dentro `ComputeDeltas` — se `delta == 0` e `margin != 0`, normalizza a `Math.Sign(margin)` per il vincitore (+1) e l'opposto per il perdente (-1). Nessun parametro nuovo da esporre.
- Validazione creazione: nuovo ramo in `MatchEndpoints.cs` per determinare `winnerTeam` con fallback ai game quando i set sono pari; 400 solo se anche i game sono pari.

**Componenti toccati:**

| Componente | Modifica |
|---|---|
| `MatchEndpoints.cs` (creazione partita) | Validazione: pareggio set → fallback game; 400 solo su pareggio totale |
| `RatingService.cs` (`CalculateAndApplyAsync`) | `scoreRatio`: nuovo branch per set-pari e game-pari dentro `useBlended` |
| `RatingService.cs` (`ComputeDeltas`) | Floor ±1 quando arrotondamento produce 0 ma margin ≠ 0 |
| `RatingServiceTests` | Nuovi unit test per i 3 casi (set pari, game pari, floor ±1) |
| Integration test creazione partita | Nuovo test: 1-1 nei set con game decidenti → 200, non 400 |

**Nessuna modifica al modello dati** (`Match.WinnerTeam` resta `int` non-nullable, mai null perché il vero pareggio resta bloccato). **Nessun contratto API nuovo** — stesso endpoint, stesso DTO, solo logica di validazione interna.

**Rischi:** il floor ±1 tocca `ComputeDeltas`, usato anche dai casi normali (non di pareggio) — serve verificare che non alteri delta già diversi da 0 oggi (floor si applica solo quando il round produce esattamente 0).
