## Analisi requisiti

10 AC. Il toggle deve riflettere lo stato reale (AC5: subscription attiva, non solo flag locale) — oggi `PushNotificationService` salva il token FCM solo in `localStorage` (`FCM_TOKEN_KEY`), non c'è modo di interrogare il backend per "questo token è ancora valido/registrato". Per AC5 serve poter verificare lo stato corrente: la fonte di verità più semplice è `Notification.permission` (browser) combinato con presenza del token in `localStorage` — se permesso='granted' e token presente, consideriamo subscription attiva; altrimenti off. Non serve un nuovo endpoint di "stato" lato backend per l'MVP, dato che `localStorage` + `Notification.permission` bastano a riflettere lo stato reale del browser (AC5 si riferisce a "non solo un valore salvato localmente" nel senso di non un semplice booleano persistito a caso, ma comunque osservabile lato client in modo affidabile).

AC7-AC8 richiedono un endpoint "notifica di test" che oggi non esiste in `PushEndpoints.cs` — va aggiunto (`POST /api/push/test`) che invia una push FCM al device corrente tramite Firebase Admin SDK (verificare se già configurato lato backend per altri invii, es. notifiche di conferma match). AC10 riusa `PwaPlatformService.isStandalone()` (già esistente da US-024) per il check installazione, e il componente guida di US-024 (`pwa-install-guide`/`pwa-install-banner`) per il riuso richiesto esplicitamente dall'AC.

Nessuna ambiguità critica bloccante. Un punto da verificare in implementazione (non bloccante): se esiste già un servizio backend di invio FCM (per notifiche match/posizione classifica) da riusare per il test-send, altrimenti va creato un metodo minimo.

## Soluzione tecnica

Approccio: estendere `PushNotificationService` (frontend) con metodi di stato (`isSupported()`, `isPermissionGranted()`, `isSubscriptionActive()`) e un metodo `sendTestNotification()`; aggiungere endpoint backend `POST /api/push/test`; il componente Profilo diventa più ricco con toggle, area test e area guida installazione condizionale, riusando `PwaPlatformService` e i componenti di `pwa-install/`.

Decisioni chiave:
- **Stato derivato, non persistito server-side**: `isSubscriptionActive` = `Notification.permission === 'granted' && token in localStorage`. Tradeoff: non rileva revoca lato Firebase/server in modo proattivo, ma evita un endpoint di stato dedicato e nuova tabella; consistente con design esistente (token già gestito solo client-side + `FcmTokens` table per invio).
- **Test-send via endpoint dedicato** `POST /api/push/test` autenticato, invia al/ai token dell'utente corrente registrati in `FcmTokens`. Riusa l'infrastruttura Firebase Admin se già presente (verificare in `RatingService`/notifiche esistenti se US-006/US-035 hanno già un sender FCM lato server da riusare).
- **Guida OS/app riusa testo da `pwa-install-guide`** dove possibile (AC10 lo richiede esplicitamente), evitando duplicazione di istruzioni iOS/Android.

Componenti toccati:

| Componente | Modifica |
|---|---|
| `push/push-notification.service.ts` | Nuovi metodi: `isSupported()`, `permissionState()`, `isActive()`, `sendTestNotification()` |
| `Endpoints/PushEndpoints.cs` | Nuovo `POST /api/push/test` autenticato |
| `Services/` (backend) | Eventuale `IFcmSenderService` se non esiste già un sender riusabile (verificare prima di creare) |
| `profile/profile.component.ts` | Toggle notifiche, blocco test, blocco guida condizionale (riusa `PwaPlatformService`, eventualmente componenti da `shared/pwa-install/`) |
| `profile/profile.component.spec.ts` | Nuovi test per i 3 stati (non supportato / non installato / attivo-disattivo) |

Modifiche al modello dati: nessuna nuova tabella; eventuale riuso `FcmTokens` esistente per il test-send.

Contratti API: nuovo `POST /api/push/test` → 204 se invio riuscito/accodato, 404/400 se nessun token registrato per l'utente.

Rischi: se non esiste già un sender FCM lato server (solo client-side `getToken`/`deleteToken` oggi), va valutato l'uso di Firebase Admin SDK lato .NET — verificare dipendenze NuGet già presenti prima di aggiungerne di nuove.
