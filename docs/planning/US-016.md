## Analisi requisiti

La storia sposta la fonte di verità degli sport da `SportsConfig` (classe statica in `src/Golp.Api/Services/SportsConfig.cs`) a una tabella `Sports` su DB. Punti di consumo attuali individuati via grep: `CircleEndpoints.cs:33` (`GET /sports` → `SportsConfig.GetAll()`), `CircleEndpoints.cs:81` (validazione sport in creazione circolo → `SportsConfig.GetBySport`), `RatingService.cs:105` (lookup sport per calcolo ELO pesato su set, introdotto in US-012/US-034). Tutti e tre vanno ridiretti a una lettura da DB.

Edge case: la migration di seed deve essere idempotente (AC esplicito) — usare `HasData` in EF (eseguita solo se assente) oppure controllo `IF NOT EXISTS` se si preferisce SQL raw; con `HasData` in `OnModelCreating` l'idempotenza è garantita dal motore di migration stesso. Il campo `Key` è confermato solo display/lookup (Open question già chiusa nel backlog) — nessuna FK da `Circle.Sport` a `Sports.Id`, quindi nessuna migrazione di dati esistenti su `Circle`/`Match` necessaria, solo popolamento della nuova tabella.

Requisito implicito: poiché `RatingService` usa `SportDto.SetWeight` per il peso ELO sui set (US-012/034), la tabella deve includere quella colonna anche se non elencata esplicitamente negli AC — altrimenti si rompe US-012. La aggiungo come colonna nullable/default 0.0 per non deviare dal nome colonne richiesto nell'AC, mappandola separatamente nel DTO di lettura. Nessuna ambiguità critica bloccante: procedo.

## Soluzione tecnica

Approccio: introdurre entità `Sport` mappata su tabella `Sports`, un repository/servizio sottile (`SportsService` con cache in-memory invalidata raramente, o lettura diretta DbContext) che sostituisce le chiamate statiche. `SportsConfig` viene eliminato e tutte le tre call site migrate a `ISportsService`/DbContext. Migration EF con `HasData` per seed idempotente dai 4 sport esistenti.

Decisioni chiave:
- **Niente cache distribuita**: dato il volume (pochi sport) e la frequenza di lettura, query diretta a ogni richiesta è accettabile; evita invalidazione complessa. Tradeoff: più query DB, trascurabile per MVP.
- **`SportDto` di servizio resta record** con stessa shape pubblica (`Sport, PointUnit, Sets, TeamSize, SetWeight`) per non toccare il contratto `GET /sports` consumato dal frontend (AC: nessuna modifica al contratto API).
- **Rimozione hard di `SportsConfig`** (non solo deprecazione) per evitare drift fra le due fonti — AC lo permette ("rimosso o deprecato").

Componenti toccati:

| Componente | Modifica |
|---|---|
| `Data/Entities/Sport.cs` (nuovo) | Entità EF: `Id, Key, DisplayName, PointUnit, Sets, TeamSize, IsActive, SetWeight` |
| `Data/AppDbContext.cs` | `DbSet<Sport> Sports`, config in `OnModelCreating`, `HasData` seed |
| `Services/SportsService.cs` (nuovo) | Sostituisce `SportsConfig`: `GetAllAsync()`, `GetBySportAsync(key)` da DB, filtra `IsActive` |
| `Services/SportsConfig.cs` | Rimosso |
| `Endpoints/CircleEndpoints.cs:33,81` | Usa `ISportsService` iniettato invece di chiamata statica |
| `Services/RatingService.cs:105` | Usa `ISportsService` (richiede metodo sync o await a monte) |
| `Program.cs` | DI registration `ISportsService` |
| Migration EF | `dotnet ef migrations add AddSportsTable` |

Modifiche al modello dati: nuova tabella `Sports` (Id PK identity, Key univoco, DisplayName, PointUnit, Sets bool, TeamSize int, IsActive bool default true, SetWeight double default 0.0).

Contratti API: nessuna modifica — `GET /sports` stessa shape risposta, filtrata server-side per `IsActive=true`.

Rischi: `RatingService.GetBySport` è chiamato in path sincrono di calcolo rating — se si introduce accesso a DB lì serve verificare che il metodo chiamante sia già async (altrimenti richiede refactor a catena). Da verificare in implementazione.
