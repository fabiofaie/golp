## Soluzione tecnica

**Architect**

**Approccio:** nuovo endpoint `POST /circles/join-by-token`, nuova rotta Angular `/join` (pubblica), e modifiche minime ai flussi di register/login per consumare il token post-auth.

**Decisioni chiave:**
- Il path `/join` Ă¨ senza authGuard: l'utente non Ă¨ autenticato quando arriva. La rotta Ă¨ aggiunta in `app.routes.ts` senza `canActivate`.
- `POST /circles/join-by-token` (autenticato): il token non va nel path perchĂ© il circolo non Ă¨ noto al chiamante; va nel body. Endpoint separato da `POST /circles/{id}/join` per non inquinare il flusso esistente.
- Register e login: logica post-auth minimale â€” se `inviteToken` in query params, fire-and-forget `joinByToken`, poi naviga al circolo anzichĂ© al dashboard.
- Conflitto di path: `/circles/join-by-token` non confligge con `/{id:guid}/join` grazie al constraint `:guid`.

Body: { inviteToken: "abc123..." }
â†’ 200: { circleId: "...", myRating: 1000, alreadyMember: false }

**Rischi:**
- Register e login devono gestire errore del join (es. token scaduto nel frattempo) senza bloccare l'autenticazione: in caso di errore del join, navigano al dashboard con un messaggio, non in loop.
