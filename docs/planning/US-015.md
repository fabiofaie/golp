## Analisi requisiti

**Analyst**

Il flusso principale è: nuova persona riceve link WhatsApp → apre `/join?token=X` → si registra → è automaticamente nel circolo. Tre sotto-casi per lo stato di autenticazione all'arrivo su `/join`:

1. **Non autenticato, non ha account** → link a `/register?inviteToken=X`
2. **Non autenticato, ha già un account** → link a `/login?inviteToken=X`
3. **Già autenticato** → join immediato via API → redirect al circolo

Il componente `/join` mostra entrambe le opzioni (registrati / hai già un account?) quando non autenticato: questo copre i casi 1 e 2 senza dover inferire se l'utente ha un account.

**Nota su AC3** ("Se l'utente è già autenticato, viene reindirizzato a `/login?inviteToken`"): questo AC è errato per il caso 3 — se già autenticato, il redirect a login è privo di senso. Il piano implementa il comportamento corretto: se autenticato, chiama direttamente `POST /circles/join-by-token` e reindirizza al circolo.

**AC4 — Nuovo endpoint backend** `POST /circles/join-by-token` con body `{ inviteToken }`:
- Cerca `Circle` dove `JoinCode == inviteToken`; 404 se non trovato
- Verifica che l'utente non sia già membro; se lo è, risponde 200 con messaggio informativo (AC7), non errore
- Crea `CircleMembership` con rating 1000, ritorna `{ circleId, myRating }`
- **Non** usa `POST /circles/{id}/join` esistente (quello richiede conoscere l'ID del circolo)

**AC5 — Redirect dopo join:** il backend restituisce `circleId`; il frontend naviga a `/circles/{circleId}/leaderboard` come landing.

**Registro + Login con inviteToken:** i componenti Angular di register e login devono già leggere `inviteToken` da `ActivatedRoute.queryParams`. Dopo un'autenticazione riuscita, se `inviteToken` è presente, chiamano `joinByToken(token)` prima di navigare al dashboard.

**Open question risolta:** token valido ma circolo eliminato → `Circle` non trovato per `JoinCode` → 404 → frontend mostra "Link non valido o scaduto".

## Soluzione tecnica

**Architect**

**Approccio:** nuovo endpoint `POST /circles/join-by-token`, nuova rotta Angular `/join` (pubblica), e modifiche minime ai flussi di register/login per consumare il token post-auth.

**Decisioni chiave:**
- Il path `/join` è senza authGuard: l'utente non è autenticato quando arriva. La rotta è aggiunta in `app.routes.ts` senza `canActivate`.
- `POST /circles/join-by-token` (autenticato): il token non va nel path perché il circolo non è noto al chiamante; va nel body. Endpoint separato da `POST /circles/{id}/join` per non inquinare il flusso esistente.
- Register e login: logica post-auth minimale — se `inviteToken` in query params, fire-and-forget `joinByToken`, poi naviga al circolo anziché al dashboard.
- Conflitto di path: `/circles/join-by-token` non confligge con `/{id:guid}/join` grazie al constraint `:guid`.

**Componenti toccati:**

| Layer | File | Modifica |
|---|---|---|
| Backend | `src/Golp.Api/Endpoints/CircleEndpoints.cs` | + `POST /join-by-token` |
| Test BE | `src/Golp.Tests/Integration/JoinByTokenEndpointTests.cs` | nuovo file |
| Service FE | `frontend/.../circles/circle.service.ts` | + `joinByToken()` |
| Componente | `frontend/.../circles/join-circle/` | nuovo (ts + html + scss) |
| Routes | `frontend/.../app.routes.ts` | + `/join` (no guard) |
| Auth FE | `frontend/.../auth/register/register.component.ts` | + post-register join |
| Auth FE | `frontend/.../auth/login/login.component.ts` | + post-login join |
| Test BE | `src/Golp.Tests/Integration/JoinByTokenEndpointTests.cs` | nuovo file |

**Contratto API:**
```
POST /circles/join-by-token
Authorization: Bearer <jwt>
Body: { inviteToken: "abc123..." }
→ 200: { circleId: "...", myRating: 1000, alreadyMember: false }
→ 200: { circleId: "...", myRating: N, alreadyMember: true }  (già membro)
→ 401: non autenticato
→ 404: { error: "Link non valido o scaduto" }
```

**Rischi:**
- Register e login devono gestire errore del join (es. token scaduto nel frattempo) senza bloccare l'autenticazione: in caso di errore del join, navigano al dashboard con un messaggio, non in loop.
